# k6 부하테스트 기록 (2026-04)

> 원칙: 수치는 측정된 그대로, 비교 조건과 한계를 함께 기록한다.

## 1. 목적

- "느릴 것 같다"가 아니라 **어디가, 왜, 얼마나** 느린지 실측
- 단일 인스턴스의 처리 한계를 확인하고 구조 개선(CQRS 등)의 도입 근거 확보

## 2. 테스트 조건

| 항목 | 내용 |
|------|------|
| 도구 | k6 (별도 EC2 t3.medium에서 실행 — 타겟과 분리) |
| 타겟 | API Gateway 경유 order-service |
| 시나리오 | reads: GET /api/orders 최대 800 RPS (600 VU) / writes: POST /api/orders 최대 150 RPS (200 VU) / details: GET /api/orders/{id} 50 RPS (80 VU) |
| 목표치 근거 | 예상 피크 트래픽 추정치 × 1.5 ≈ 800 RPS. HikariCP pool 이론 한계(750 RPS) 바로 위로 설정해 풀 고갈 여부 검증 |

## 3. 실측: HikariCP 커넥션 풀 고갈

혼합 부하(읽기 800 + 쓰기 150 RPS)에서:

```
HikariPool-1 - Connection is not available
  total=3, active=3, idle=0, waiting=161
  request timed out after 3007ms
```

| 부하 조건 | 읽기 P95 | 에러율 |
|-----------|----------|--------|
| smoke (읽기 5 + 쓰기 2 RPS) | 67ms | 0% |
| 최대 (읽기 800 + 쓰기 150 RPS) | **3,920ms (58배)** | 98.86% |

**해석**: 같은 읽기 엔드포인트가 쓰기 부하 때문에 58배 느려졌다. 읽기와 쓰기가 동일 커넥션 풀을 공유하는 구조적 문제 → 풀 크기 증가는 임시 완화일 뿐, 읽기/쓰기 분리(CQRS)가 근본 해결.

## 4. CQRS 전/후 비교

동일 혼합 부하(읽기 800 + 쓰기 150 RPS) 기준:

| 항목 | MySQL 단일 (pool=3) | CQRS — 조회 MongoDB 분리 (pool=10) |
|------|--------------------|------------------------------------|
| 읽기 P95 | 9,990ms (타임아웃) | **153ms** |
| 읽기 에러율 | 97.02% | **0.00%** |
| 테스트 시간 내 주문 생성 완료 | ~400건 | **3,535건** |

### ⚠️ 수치의 한계 (정직하게)

- 위 비교는 **pool=3 vs pool=10으로 조건이 동일하지 않다.** 공정 비교 기준(MySQL + 인덱스 + pool=10)으로는 **5~10배 개선이 현실적 수치**다.
- "3,535건"은 TPS가 아니라 **테스트 시간 내 주문 생성 완료 건수**다.
- 실제 사용자 트래픽이 아닌 합성 부하다.
- 쓰기 에러율은 잔존한다 — 원인은 order→product **동기 HTTP 체인의 연쇄 장애**로 특정했고, Circuit Breaker 도입으로 해결 예정.

## 5. 부수 실험: Virtual Threads (100 VU)

| 지표 | VT 비활성 | VT 활성 |
|------|----------|---------|
| 처리량 | 115 req/s | 105 req/s (차이는 측정 노이즈 수준) |
| 플랫폼 스레드 피크 | 121개 | **46개** |

처리량 개선이 아니라 **같은 처리량을 더 적은 OS 스레드로** 달성하는 것이 효과임을 확인.

## 6. 1차 실행 실패 기록

1차 실행은 k6의 Prometheus remote write(880 VU 메트릭 전송)가 인프라 서버 OOM을 유발해 10분 만에 조기 종료됐다. 2차 실행에서 정상 완료(총 반복 206,367회). 1차 실패의 원인 분석: 부하 생성기의 메트릭 전송 부하까지 용량 계획에 포함해야 하며, 관측 인프라와 메시징 인프라는 장애 도메인을 분리해야 한다는 교훈을 얻었다.
