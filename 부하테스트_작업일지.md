## 4. 재고 예약 — Redis 원자적 DECRBY 전환 (inventory-service)

### 문제
`SELECT FOR UPDATE` (비관적 락) 사용 시
800 RPS 부하 → 같은 상품에 동시 락 경합 → 전체 타임아웃

### 기존 흐름
```
reserve() → SELECT * FROM inventories WHERE product_id = ? FOR UPDATE
         → inventory.reserve() → save
```

### 변경 흐름
```
reserve() → Redis DECRBY inv:stock:{productId} {qty}
         → remaining < 0 이면 INCRBY 롤백 + InsufficientStockException
         → 성공 시 InventoryReservation DB 저장 (락 없음)
```

### 변경 파일
| 파일 | 내용 |
|------|------|
| `build.gradle.kts` | `spring-boot-starter-data-redis` 추가 |
| `InventoryService.java` | reserve/release/addStock Redis 연동 + Redis 키 미존재 시 DB fallback |
| `RedisStockInitializer.java` | 시작 시 DB → Redis 동기화 (PENDING 예약 차감), Redis 실패 시 try-catch로 서비스 계속 기동 |
| `InventoryReservationRepository.java` | `findByStatus()` 추가 |

### Redis 키
- `inv:stock:{productId}` = 현재 예약 가능 수량

### Redis 동기화 로직
```
시작 시: Redis 재고 = DB.availableQuantity - sum(PENDING 예약 수량)
reserve: Redis DECRBY (DB 락 없음) → 키 없으면 DB fallback 후 DECRBY
release: Redis INCRBY + DB 예약 RELEASED
confirm: DB 예약 CONFIRMED (Redis 변경 없음)
addStock: Redis INCRBY + DB availableQuantity 증가
```

## 7. 800 RPS 목표치 근거

### 비즈니스 근거 (부산+경남 2단계 런칭)
```
부산 340만 + 경남 330만 = 670만 인구
하루 1회 주문 가정 = 670만 주문/일

피크 1시간 30% 집중:
670만 × 0.3 / 3,600 = 558 주문/초

판매자 트래픽 + 안전 마진 1.5×:
558 × 1.5 ≈ 837 → 800 RPS
```

### 기술 검증 목적
```
HikariCP pool-15 이론 한계: 15 / 0.02s = 750 RPS
800 RPS = 한계 바로 위 → pool pending 발생 여부 검증
```

### 스케일아웃 기준
```
800 RPS = 1 인스턴스 한계
전국 절반(2,350 RPS) 대응 → 약 3 인스턴스 필요
```

## 8. Rate Limiting / Snowflake ID 도입 근거 (면접용)

### Rate Limiting
- 피크 수백 RPS가 결제 API(Toss 외부 호출, 건당 비용)에 집중 가능
- 악성 호출 시 비용 폭증 + Brute Force 위험
- Gateway 레벨 토큰 버킷으로 downstream 완전 보호
- **실측**: downstream 도달 100% → 0.33% (이론값 35건과 정확히 일치)

### Snowflake ID
- ECS 스케일아웃 시 여러 인스턴스 동시 INSERT → AUTO_INCREMENT 충돌 가능
- 순차 ID 노출 시 경쟁사가 하루 주문량 추정 가능 (보안)
- Snowflake: `[45-bit 타임스탬프][10-bit machineId][8-bit sequence]` → DB 없이 분산 고유 ID 발급

## 9. Debezium CDC 설정 어려움

### 정량적
| 항목 | 내용 |
|------|------|
| RDS binlog 설정 | 파라미터 그룹 변경 후 RDS 재시작 필요 (다운타임) |
| MySQL 계정 권한 | REPLICATION SLAVE + REPLICATION CLIENT + SELECT (각 테이블별) — 하나 빠지면 조용히 실패 |
| EventRouter 필드 | `route.by.field`, `field.event.id`, `field.event.payload` 3개 모두 정확해야 함 |

### 정성적
- **"조용한 실패"**: 커넥터 상태 `RUNNING`인데 이벤트 0건 — Kafka 직접 consume으로 검증 필요
- **로컬↔운영 환경 차이**: hostname, bootstrap.servers가 환경마다 달라 JSON 환경별 분리 필요
- **단일 RDS 다중 스키마**: `table.include.list: "{schema}.outbox_event"` 명시 안 하면 이벤트 섞임
