# Order Service 도메인 모델

> Aggregate Root = **Order**
> 통화 = **KRW(원화) only**
> 목표: "주문 생성/결제/완료/취소"의 **상태 전이**와 **금액/라인 불변식**을 도메인에서 강제한다.

---

## 1. 도메인 경계

- **Order Service**는 주문의 생명주기(생성 → 결제 → 배달완료 → 취소)를 책임진다.
- 외부 시스템(회원/상품/결제/배송/마트)은 별도 서비스로 가정한다.
- Order Service는 외부 식별자를 **참조만** 하며, 주문 시점 정보를 스냅샷으로 저장한다.
  - `buyerId`: auth-service Customer PK
  - `productId`: product-service PK
  - `martId`: mart-service PK (MartSnapshot에 스냅샷)
- **배송지/저장 주소는 auth-service가 관리** (SavedAddress). Order는 주문 시 선택된 주소를 DeliverySnapshot으로 복사해 저장.

---

## 2. 핵심 개념

### 2.1 Aggregate Root
- **Order (Entity / Aggregate Root)**
  주문의 상태 전이와 불변식(invariant)을 보장하는 최상위 도메인 객체.

### 2.2 Value Objects

| VO | 설명 |
|----|------|
| `Money` | 원화 금액. 불변, 값 기반 동등성 |
| `OrderLine` | 주문 항목. 주문 시점 상품명/단가 스냅샷 + 수량 |
| `DeliverySnapshot` | 주문 당시 배송지 스냅샷 (zipCode, roadAddress, detailAddress). 수령인/전화번호 미포함 |
| `MartSnapshot` | 주문 당시 마트 정보 스냅샷 (martId, martName, martPhone?) |
| `OrderMemo` | 요청사항 (orderRequest: 판매자용, deliveryRequest: 기사용) |
| `ChargeLine` | 요금 명세 항목 (SUBTOTAL, DELIVERY_FEE, DISCOUNT, POINT) |

> **수령인명/전화번호 미포함 이유**: 이벤트/로그에 개인정보 노출 방지. 필요 시 orderId로 DB 직접 조회.

## 3. 상태 모델

### 3.1 OrderStatus
```
PENDING_PAYMENT  결제 대기 (초기)
PAID             결제 완료
PAYMENT_FAILED   결제 실패
CONFIRMED        주문 완료 (배달 완료 후)
CANCELED         취소 (터미널)
```

### 3.2 상태 전이

```
PENDING_PAYMENT
    ├── → PAID            결제 성공 (payment.result.v1 DONE)
    └── → PAYMENT_FAILED  결제 실패 (payment.result.v1 FAILED)

PAYMENT_FAILED
    └── → PENDING_PAYMENT 재결제 허용

PAID
    └── → CONFIRMED       배달 완료 (delivery.completed.v1)

CONFIRMED  터미널
CANCELED   터미널
```

## 4. 엔티티/VO 정의

### 4.1 Order (Aggregate Root)

#### 속성
| 필드 | 타입 | 설명 |
|------|------|------|
| `id` | Long | DB PK (Snowflake ID) |
| `tossOrderId` | String | Toss 결제용 주문 식별자 |
| `buyerId` | Long | auth-service Customer PK |
| `payMethod` | OrderPayMethod | CARD / CASH_PREPAID / CASH_ON_DELIVERY / POINT / MIXED |
| `status` | OrderStatus | 주문 상태 |
| `orderLines` | List\<OrderLine\> | 주문 상품 목록 |
| `chargeLines` | List\<ChargeLine\> | 요금 명세 |
| `deliverySnapshot` | DeliverySnapshot | 배송지 스냅샷 |
| `martSnapshot` | MartSnapshot | 마트 정보 스냅샷 |
| `orderMemo` | OrderMemo? | 요청사항 (nullable) |
| `totalAmount` | Money | 최종 결제 금액 |
| `createdAt` | LocalDateTime | 주문 생성 시각 |
| `paidAt` | LocalDateTime? | 결제 완료 시각 |
| `confirmedAt` | LocalDateTime? | 주문 완료 시각 |
| `canceledAt` | LocalDateTime? | 취소 시각 |

#### 불변식
- orderLines는 1개 이상
- 각 라인의 quantity는 1 이상
- totalAmount = 상품소계 + 배달팁 (5만원 이상 무료)
- 결제 금액이 totalAmount와 다르면 예외

#### 도메인 행위
- `create(OrderCreateRequest)`: 주문 생성. tossOrderId 생성, chargeLines 계산. CASH_ON_DELIVERY는 즉시 PAID
- `markAsPaid(confirmedAmount)`: 결제 완료. 금액 검증 포함. 멱등
- `markPaymentFailed()`: 결제 실패. 멱등
- `markAsCompleted()`: 배달 완료 → CONFIRMED. 멱등. CASH_ON_DELIVERY는 paidAt도 함께 설정
- `confirmCashPayment()`: 현금 선불 — 판매자가 현금 수령 확인 → PAID
- `retryPayment()`: 결제 실패 후 재결제 → PENDING_PAYMENT + tossOrderId 재발급
- `cancel()`: PENDING_PAYMENT → CANCELED (터미널)

### 4.2 DeliverySnapshot

```java

```
- 수령인명/전화번호 **없음** — 로그/이벤트 개인정보 노출 방지
- 저장된 주소 목록(SavedAddress)은 auth-service가 관리
- `fullAddress()`: "(zipCode) roadAddress detailAddress" 반환

- 마트명이 나중에 변경되어도 주문내역에는 주문 당시 이름 유지

- 두 필드 모두 nullable
- 빈 문자열 입력 시 null로 정규화
- `OrderMemo.of(orderRequest, deliveryRequest)` 팩토리 메서드

### 4.5 ChargeLine

```java

```

| ChargeType | 설명 |
|------------|------|
| SUBTOTAL | 상품 소계 |

- `Money.zero()`, `Money.of(long)` 팩토리

## 5. auth-service — SavedAddress (연동 구조) ✅ 구현됨

> 저장된 배송지는 auth-service에서 관리. order-service는 주문 시 스냅샷만 저장.

```
auth-service:
  Customer
   └── List<SavedAddress>
         - id (Snowflake), zipCode, roadAddress, detailAddress, label(집/회사 등), isDefault
         - customer_id (ManyToOne → Customer)

API (X-User-Id 헤더로 customerId 식별 — Gateway JWT uid claim 주입):
  GET    /auth/customers/addresses          저장된 주소 목록
  POST   /auth/customers/addresses          주소 추가 (isDefault=true 시 기존 기본주소 자동 해제)
  PUT    /auth/customers/addresses/{id}     주소 수정
  DELETE /auth/customers/addresses/{id}     주소 삭제
```

**Member(판매자) 변경사항**:
- PK: `email` → `memberId` (Long, Snowflake ID)
- JWT `uid` claim = `memberId` → Gateway `X-User-Id` 주입 → `martId`로 사용 (mart-service 도입 전까지)

주문 생성 흐름:
1. 프론트가 auth-service에서 주소 목록 조회
2. 소비자가 원하는 주소 선택
3. 선택한 주소의 zipCode/roadAddress/detailAddress를 OrderCreateRequest에 포함
4. order-service가 DeliverySnapshot으로 스냅샷 저장

## 6. 유스케이스

### UC-01 주문 생성
- 입력: buyerId(Gateway 주입), payMethod, orderLines, deliverySnapshot, martSnapshot, orderMemo?
- 흐름:
  1. product-service 가격 검증 (동기 HTTP)
  2. inventory-service 재고 예약 (동기 HTTP)
  3. Order.create() → chargeLines 자동 계산
  4. DB 저장

### UC-02 결제 완료
- Kafka `payment.result.v1` (status=DONE) 수신
- Order.markAsPaid(confirmedAmount) — 금액 검증
- OutboxEvent 저장 → order.paid.v1 발행 → delivery-service 배송 생성

### UC-03 결제 실패
- Kafka `payment.result.v1` (status=FAILED) 수신
- Order.markPaymentFailed()
- inventory-service 재고 해제

### UC-04 배달 완료
- Kafka `delivery.completed.v1` 수신
- Order.markAsCompleted() → CONFIRMED
- CASH_ON_DELIVERY인 경우 paidAt도 함께 설정

### UC-05 현금 선불 확인
- 판매자 앱 → `PATCH /api/orders/{id}/confirm-cash` (X-User-Type: MEMBER)
- Order.confirmCashPayment() → PAID
- OutboxEvent 저장 → order.paid.v1 발행 → delivery-service 배송 생성

### UC-06 재결제
- 소비자 앱 → `PATCH /api/orders/{id}/retry-payment`
- Order.retryPayment() → PENDING_PAYMENT + tossOrderId 재발급
- 소비자가 새 tossOrderId로 Toss 결제 재시도

## 7. JPA 매핑 전략

- `Order`: `@Entity`, `@Table(name="orders")`
- VO들: `@Embeddable` record
- `orderLines`: `@ElementCollection` + `@CollectionTable(name="order_lines")`
- `chargeLines`: `@ElementCollection` + `@CollectionTable(name="charge_lines")`
- `DeliverySnapshot`, `MartSnapshot`, `OrderMemo`: `@Embedded` (Order 테이블에 컬럼으로 포함)
