# Order Service 도메인 모델

> Aggregate Root = **Order**
> 통화 = **KRW(원화) only**
> 목표: "주문 생성/결제/완료/취소"의 **상태 전이**와 **금액/라인 불변식**을 도메인에서 강제한다.

---

## 1. 도메인 경계

- **Order Service**는 주문의 생명주기(생성 → 결제 → 배달완료 → 취소)를 책임진다.
- 외부 시스템(회원/상품/결제/배송/마트)은 별도 서비스로 가정한다.
- Order Service는 외부 식별자를 **참조만** 하며, 주문 시점 정보를 스냅샷으로 저장한다.
  - `buyerId`: auth-service Customer PK
  - `productId`: product-service PK
  - `martId`: mart-service PK (MartSnapshot에 스냅샷)
- **배송지/저장 주소는 auth-service가 관리** (SavedAddress). Order는 주문 시 선택된 주소를 DeliverySnapshot으로 복사해 저장.

---

## 2. 핵심 개념

### 2.1 Aggregate Root
- **Order (Entity / Aggregate Root)**
  주문의 상태 전이와 불변식(invariant)을 보장하는 최상위 도메인 객체.

### 2.2 Value Objects

| VO | 설명 |
|----|------|
| `Money` | 원화 금액. 불변, 값 기반 동등성 |
| `OrderLine` | 주문 항목. 주문 시점 상품명/단가 스냅샷 + 수량 |
| `DeliverySnapshot` | 주문 당시 배송지 스냅샷 (zipCode, roadAddress, detailAddress). 수령인/전화번호 미포함 |
| `MartSnapshot` | 주문 당시 마트 정보 스냅샷 (martId, martName, martPhone?) |
| `OrderMemo` | 요청사항 (orderRequest: 판매자용, deliveryRequest: 기사용) |
| `ChargeLine` | 요금 명세 항목 (SUBTOTAL, DELIVERY_FEE, DISCOUNT, POINT) |

> **수령인명/전화번호 미포함 이유**: 이벤트/로그에 개인정보 노출 방지. 필요 시 orderId로 DB 직접 조회.

## 3. 상태 모델

### 3.1 OrderStatus
```
PENDING_PAYMENT  결제 대기 (초기)
PAID             결제 완료
PAYMENT_FAILED   결제 실패
CONFIRMED        주문 완료 (배달 완료 후)
CANCELED         취소 (터미널, 🔜 미구현)
```

### 3.2 상태 전이

```
PENDING_PAYMENT
    ├── → PAID            결제 성공 (payment.result.v1 DONE)
    │                     또는 CASH_ON_DELIVERY 생성 시 즉시
    │                     또는 CASH_PREPAID confirm-cash 확인 시
    └── → PAYMENT_FAILED  결제 실패 (payment.result.v1 FAILED)

PAYMENT_FAILED
    └── → PENDING_PAYMENT 재결제 허용 (tossOrderId 재발급)

PAID
    └── → CONFIRMED       배달 완료 (delivery.completed.v1)

CONFIRMED  터미널
CANCELED   터미널 (🔜 미구현 — PENDING_PAYMENT 상태만 허용 예정)
```

## 4. 엔티티/VO 정의

### 4.1 Order (Aggregate Root)

#### 속성
| 필드 | 타입 | 설명 |
|------|------|------|
| `id` | Long | DB PK (Snowflake ID) |
| `tossOrderId` | String | Toss 결제용 주문 식별자 (`ORD_{timestamp}_{uuid8}`) |
| `buyerId` | Long | auth-service Customer PK |
| `payMethod` | OrderPayMethod | CARD / CASH_PREPAID / CASH_ON_DELIVERY / POINT / MIXED |
| `status` | OrderStatus | 주문 상태 |
| `orderLines` | List\<OrderLine\> | 주문 상품 목록 |
| `chargeLines` | List\<ChargeLine\> | 요금 명세 (SUBTOTAL, DELIVERY_FEE) |
| `deliverySnapshot` | DeliverySnapshot | 배송지 스냅샷 |
| `martSnapshot` | MartSnapshot | 마트 정보 스냅샷 |
| `orderMemo` | OrderMemo? | 요청사항 (nullable) |
| `totalAmount` | Money | 최종 결제 금액 (상품소계 + 배달팁) |
| `createdAt` | LocalDateTime | 주문 생성 시각 |
| `paidAt` | LocalDateTime? | 결제 완료 시각 |
| `confirmedAt` | LocalDateTime? | 주문 완료 시각 |
| `canceledAt` | LocalDateTime? | 취소 시각 |

#### 불변식
- orderLines는 1개 이상
- 각 라인의 quantity는 1 이상
- totalAmount = 상품소계 + 배달팁 (5만원 이상 무료, 미만 3,000원)
- 결제 금액이 totalAmount와 다르면 예외

#### 도메인 행위
| 메서드 | 전이 | 설명 |
|--------|------|------|
| `create(req)` | → PENDING_PAYMENT 또는 PAID | tossOrderId 생성, chargeLines 계산. CASH_ON_DELIVERY는 즉시 PAID |
| `markAsPaid(confirmedAmount)` | PENDING_PAYMENT → PAID | 금액 검증 포함. 멱등 |
| `markPaymentFailed()` | PENDING_PAYMENT → PAYMENT_FAILED | 멱등. 이미 PAID면 무시 |
| `markAsCompleted()` | PAID → CONFIRMED | 멱등. CASH_ON_DELIVERY는 paidAt도 함께 설정 |
| `confirmCashPayment()` | PENDING_PAYMENT → PAID | CASH_PREPAID 전용. 판매자가 현금 수령 확인 |
| `retryPayment()` | PAYMENT_FAILED → PENDING_PAYMENT | tossOrderId 재발급 |

### 4.2 DeliverySnapshot

```java
record DeliverySnapshot(String zipCode, String roadAddress, String detailAddress)
```
- 우편번호: 5자리 숫자만 허용 (`^\d{5}$`)
- 수령인명/전화번호 **없음** — 로그/이벤트 개인정보 노출 방지
- `fullAddress()`: "(zipCode) roadAddress detailAddress" 반환

---

### 4.3 MartSnapshot

```java
record MartSnapshot(Long martId, String martName, String martPhone)
```
- 마트명이 나중에 변경되어도 주문내역에는 주문 당시 이름 유지
- martPhone nullable
- **mart-service 없이도 주문 생성 가능** (요청 body에 직접 포함, DB 검증 없음)

---

### 4.4 OrderMemo

```java
record OrderMemo(String orderRequest, String deliveryRequest)
```
- 두 필드 모두 nullable
- 빈 문자열 입력 시 null로 정규화

---

### 4.5 ChargeLine

```java
record ChargeLine(ChargeType type, Money amount)
```

| ChargeType | 설명 |
|------------|------|
| SUBTOTAL | 상품 소계 |
| DELIVERY_FEE | 배달팁 (5만원 이상 0원) |
| DISCOUNT | 할인쿠폰 (🔜 미구현) |
| POINT | 포인트 사용 (🔜 미구현) |

---

### 4.6 Money

- `amount`: long (0 이상)
- `plus(Money)`, `multiply(int)` 연산 (오버플로우 방지: `Math.addExact`, `Math.multiplyExact`)
- `Money.zero()`, `Money.of(long)` 팩토리
- JSON: `@JsonValue`로 amount 값만 직렬화 (예: `15000`, `{"amount":15000}` 아님)

---

### 4.7 OrderLine

| 필드 | 타입 | 검증 |
|------|------|------|
| `productId` | Long | not null |
| `productNameSnapshot` | String | not blank, 200자 이하 |
| `unitPrice` | Money | 0 이상 |
| `quantity` | int | 1 이상 |

- `lineAmount()`: unitPrice × quantity

## 5. auth-service — SavedAddress (연동 구조)

> 저장된 배송지는 auth-service에서 관리. order-service는 주문 시 스냅샷만 저장.

```
auth-service:
  Customer
   └── List<SavedAddress>
         - id (Snowflake), zipCode, roadAddress, detailAddress, label(집/회사 등), isDefault
         - customer_id (ManyToOne → Customer)

API (X-User-Id 헤더로 customerId 식별 — Gateway JWT uid claim 주입):
  GET    /auth/customers/addresses          저장된 주소 목록
  POST   /auth/customers/addresses          주소 추가
  PUT    /auth/customers/addresses/{id}     주소 수정
  DELETE /auth/customers/addresses/{id}     주소 삭제
```

주문 생성 흐름:
1. 프론트가 auth-service에서 주소 목록 조회
2. 소비자가 원하는 주소 선택
3. 선택한 주소의 zipCode/roadAddress/detailAddress를 OrderCreateRequest에 포함
4. order-service가 DeliverySnapshot으로 스냅샷 저장

## 6. REST API

> **CQRS 적용**: 읽기(GET)는 order-query-service(포트 8092, MongoDB)로 분리. order-service는 쓰기(Command)만 담당.

| Method | Path | 권한 | 설명 |
|--------|------|------|------|
| POST | `/api/orders` | CUSTOMER | 주문 생성 |
| PATCH | `/api/orders/{orderId}/confirm-cash` | MEMBER | 현금 선불 수령 확인 → PAID |
| PATCH | `/api/orders/{orderId}/retry-payment` | CUSTOMER (본인) | 결제 실패 후 재결제 → PENDING_PAYMENT |

### 에러 응답 형식

```json
{ "code": "ORDER_NOT_FOUND",  "message": "주문을 찾을 수 없습니다: orderId=123" }  // 404
{ "code": "BAD_REQUEST",      "message": "가격이 변경된 상품이 있습니다." }           // 400
{ "code": "INVALID_STATE",    "message": "본인의 주문만 재결제할 수 있습니다." }       // 409
{ "code": "VALIDATION_ERROR", "message": "orderLines: must not be empty" }         // 400
{ "code": "INTERNAL_ERROR",   "message": "서버 오류가 발생했습니다." }                // 500
```

## 7. 유스케이스

### UC-01 주문 생성
- 입력: buyerId(Gateway X-User-Id 헤더 주입), payMethod, orderLines, deliverySnapshot, martSnapshot, orderMemo?
- 흐름:
  1. product-service 가격 검증 (동기 HTTP) — 실패 시 주문 생성 중단
  2. inventory-service 재고 예약 (동기 HTTP) — 실패(재고 부족) 시 주문 생성 중단
  3. Order.create() → chargeLines 자동 계산
  4. DB 저장
  5. CASH_ON_DELIVERY인 경우: Outbox 저장(order.paid.v1) + 재고 확정

### UC-02 결제 완료
- Kafka `payment.result.v1` (status=DONE) 수신
- Order.markAsPaid(confirmedAmount) — 금액 검증
- OutboxEvent 저장 → order.paid.v1 발행 → delivery-service 배송 생성
- inventory-service 재고 확정 (RESERVED → DEDUCTED)

### UC-03 결제 실패
- Kafka `payment.result.v1` (status=FAILED) 수신
- Order.markPaymentFailed()
- inventory-service 재고 해제 (RESERVED → AVAILABLE)

### UC-04 배달 완료
- Kafka `delivery.completed.v1` 수신
- Order.markAsCompleted() → CONFIRMED
- CASH_ON_DELIVERY인 경우 paidAt도 함께 설정 (배달 완료 = 실제 결제 시점)

### UC-05 현금 선불 확인
- 판매자 앱 → `PATCH /api/orders/{id}/confirm-cash` (X-User-Type: MEMBER)
- Order.confirmCashPayment() → PAID
- OutboxEvent 저장 → order.paid.v1 발행 → delivery-service 배송 생성
- inventory-service 재고 확정

### UC-06 재결제
- 소비자 앱 → `PATCH /api/orders/{id}/retry-payment` (X-User-Id 헤더 필수)
- **buyerId 소유권 검증** — 본인 주문이 아닌 경우 409
- Order.retryPayment() → PENDING_PAYMENT + tossOrderId 재발급
- 소비자가 새 tossOrderId로 Toss 결제 재시도

## 8. JPA 매핑 전략

- `Order`: `@Entity`, `@Table(name="orders")`
- VO들: `@Embeddable` record
- `orderLines`: `@ElementCollection` + `@CollectionTable(name="order_lines")`
- `chargeLines`: `@ElementCollection` + `@CollectionTable(name="charge_lines")`
- `DeliverySnapshot`, `MartSnapshot`, `OrderMemo`: `@Embedded` (Order 테이블에 컬럼으로 포함)
- PK 전략: Snowflake ID (45bit 타임스탬프, 8bit 일련번호)

## 9. 확장 포인트

- 주문 취소 (PENDING_PAYMENT만 허용, pay-service 환불 트리거, `order.canceled.v1` 발행)
- 할인쿠폰/포인트 적용 → ChargeLine에 DISCOUNT/POINT 추가
- ElasticSearch CQRS 검색 분리 (search-service)
- point-service 연동: delivery.completed.v1 수신 → 포인트 적립
