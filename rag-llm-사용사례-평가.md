## 평가 프레임워크

### 핵심 질문 3가지

| # | 질문 | 판단 기준 |
|---|------|-----------|
| Q1 | 학습 데이터에 없는 정보가 필요한가? | 우리 마트 상품 목록, 주문 이력 등 LLM이 모르는 도메인 데이터 필요 여부 |
| Q2 | 최신 정보나 자주 업데이트되는 정보가 필요한가? | 실시간 재고, 가격 변동, 주문 상태 등 동적 데이터 필요 여부 |
| Q3 | 정보의 인용/출처 기반 생성이 필요한가? | 사용자에게 "이 상품 기준으로 추천드려요" 같은 출처 제공이 도움이 되는가 |

### 유의사항 (도입 전 반드시 확인)

```
1. 단순히 프롬프트 엔지니어링으로 해결 가능한가?
2. LLM에 컨텍스트를 직접 전달하는 것만으로 충분한가?
   → YES면 RAG 불필요, 직접 전달로 해결
```

### RAG 필요/불필요 기준

**RAG가 필요한 경우**
- 정보가 실시간으로 업데이트되는 경우
- 도메인에 특화된 내부 지식이 필요한 경우

**RAG가 불필요한 경우**
- 간단한 텍스트 처리
- 정적이고 변하지 않는 데이터
- 일반적인 대화

## 사용사례

### Case 1. 대화형 음성 주문 플로우

> 마트 직원처럼 상품 추천부터 결제/배송지 확인까지 한 대화 안에서 완결되는 주문 경험

#### 전체 대화 흐름

```
[소비자] "아이 간식으로 달달한 거 뭐가 좋아요?"
[AI]     "제주 감귤(15,000원)이랑 꿀사과(12,000원) 있어요. 감귤이 요즘 많이 찾으세요."
[소비자] "감귤로 주세요"
[AI]     "감귤은 1박스(10개) 단위예요. 몇 박스 드릴까요?"
[소비자] "저번이랑 똑같이요"
[AI]     "지난번에 2박스 주문하셨네요. 2박스(30,000원)로 할까요?"
[소비자] "네"
[AI]     "결제는 어떻게 하실 건가요? 카드/현금 중 선택해 주세요."
[소비자] "카드요"
[AI]     "배송지는 지난번이랑 같은 서울시 강남구로 할까요?"
[소비자] "네 맞아요"
[AI]     "제주 감귤 2박스, 30,000원 카드결제로 주문할게요. 맞으시죠?"
[소비자] "네"
[주문 생성 → POST /api/orders]
```

#### 대화 턴별 기술 분류

| 대화 단계 | 소비자 발화 | 필요한 기술 | 이유 |
|-----------|------------|------------|------|
| 상품 추천 | "달달한 간식 뭐 있어요?" | **RAG** | 우리 마트 상품 목록은 LLM 학습 데이터에 없음. 전체 목록을 프롬프트에 넣으면 토큰 폭발 → ES 검색 후 관련 상품만 주입 |
| 주문 의도 파악 | "감귤로 주세요" | **Prompt Engineering** | 일반 언어 이해 범위. 컨텍스트 직접 전달로 충분 |
| 단위 모호성 해소 | "몇 박스요?" | **Function Calling** | 상품 단위 정보(unit: "박스", unitSize: 10)를 product-service에서 조회 |
| 이전 주문 참조 | "저번이랑 똑같이요" | **Function Calling** | 특정 사용자의 최근 주문 이력 정확 조회. 벡터 유사도 검색이 아닌 API 직접 호출 |
| 결제 방식 선택 | "카드요" | **Prompt Engineering** | 정해진 선택지(카드/현금) 중 의도 파악만 필요 |
| 배송지 확인 | "지난번이랑 같은 곳" | **Function Calling** | 이전 배송지 조회 → order-query-service API 호출 |
| 최종 확인/주문 | "네" | **Prompt Engineering** | 확인 의도 파악 후 기존 POST /api/orders 호출 |

#### 평가

| 질문 | 판단 | 근거 |
|------|------|------|
| Q1. 학습 데이터에 없는 정보 필요? | **YES** | 우리 마트 상품 목록, 주문 이력, 배송지 이력 모두 LLM 학습 데이터에 없음 |
| Q2. 최신/자주 업데이트되는 정보 필요? | **YES** | 상품 재고/가격 실시간 변동, 주문 이력도 매 주문마다 갱신 |
| Q3. 출처 기반 생성 필요? | **YES** | "저희 마트 제주 감귤 기준", "지난 주문 기준"처럼 근거 있는 답변이 신뢰성 핵심 |

#### 프롬프트 엔지니어링만으로 해결 가능한가?

```
NO — 두 가지 이유

1. 상품 추천:
   전체 상품 목록을 프롬프트에 직접 넣으면
   상품 1,000개 × 설명 = 토큰 폭발 (비용 + 컨텍스트 한계)
   → 관련 상품만 골라서 주입하는 검색 단계(RAG) 필수

2. 주문 이력 / 배송지:
   특정 사용자의 데이터는 프롬프트에 미리 넣을 수 없음
   → 대화 중 실시간으로 API 조회(Function Calling) 필수
```

#### 결론

```
✅ RAG 필요      → 상품 추천 단계
✅ Function Calling 필요 → 주문 이력, 배송지 조회 단계
✅ Prompt Engineering   → 의도 파악, 선택지 처리 단계

세 가지 기술이 한 대화 플로우 안에서 역할을 나눠 동작
```

## 기술 역할 분담 정리

```
RAG (Retrieval-Augmented Generation)
  담당: 상품 추천
  흐름: 질문 임베딩 → ES kNN 벡터 검색 → 관련 상품 Top 5 프롬프트 주입 → 추천 생성
  왜 RAG?: "비슷한 것 찾기" — 자연어 질문과 의미적으로 가까운 상품 검색

Function Calling (Tool Use)
  담당: 주문 이력 조회, 상품 단위 조회, 배송지 이력 조회
  흐름: LLM이 필요 시점에 API 직접 호출 → 결과를 대화에 반영
  왜 Function Calling?: "정확한 것 가져오기" — orderId/buyerId 기준 단건 조회

Prompt Engineering
  담당: 의도 파악, 수량/결제수단 추출, 최종 확인 처리
  흐름: 시스템 프롬프트 + 대화 이력 → LLM이 다음 액션 결정
  왜 Prompt Engineering?: 정해진 선택지나 일반 언어 이해는 검색 불필요
```

## RAG 프레임워크 결정

> **유의사항**: 프레임워크의 기술적 장점보다 익숙한 환경이 생산성에 더 큰 영향을 미침.
> 모든 항목에서 완벽한 선택지는 없으므로, 이 프로젝트에서 중요한 것의 우선순위를 먼저 설정.

### 이 프로젝트 우선순위

```
1순위: 단기간 습득 가능한가  (포트폴리오, 혼자 개발, 빠른 구현 목표)
2순위: 요구사항 변경 시 유연한가  (대화 플로우가 설계 중 바뀔 수 있음)
3순위: 로깅/추적 가능한가  (RAG 품질 디버깅, 면접 시 "어떻게 개선했나" 설명용)
```

### 비교표: Spring AI vs LangChain4j

> LangChain (Python) 은 별도 Python 서버가 필요해 이 프로젝트(Java/Spring) 구조에 맞지 않아 제외.
> Java 기반 두 선택지만 비교.

| 항목 | Spring AI | LangChain4j |
|------|-----------|-------------|
| **언어/생태계** | Java, Spring 네이티브 | Java, 독립 라이브러리 |
| **성숙도** | 낮음 (1.0 GA 2024년) | 중간 (Python LangChain 포팅) |
| **학습 자료** | 적음, 한국어 거의 없음 | 중간, Python 자료 참고 가능 |
| **Spring Boot 통합** | ✅ 완벽 (자동 설정, DI, yml 설정) | △ 별도 Bean 등록 필요 |
| **RAG 추상화** | 기본 제공 (VectorStore, Advisor) | 풍부 (Chain, Agent, Memory 패턴) |
| **Function Calling** | ✅ `@Tool` 애너테이션 | ✅ `@Tool` 애너테이션 |
| **스트리밍** | ✅ Flux 기반 | ✅ 지원 |
| **지원 LLM** | Claude, GPT, Gemini 등 | Claude, GPT, Gemini 등 |
| **벡터 DB 지원** | ES, Redis, Pinecone 등 | ES, Redis, Pinecone 등 |
| **로깅/추적** | Micrometer + Actuator (이미 프로젝트에 있음) | 자체 로깅 + LangSmith 별도 연동 |
| **요구사항 변경 유연성** | 중간 (추상화 얇아 직접 제어 쉬움) | 높음 (체인/에이전트 패턴 풍부) |
| **단기간 습득** | ✅ Spring 개발자라면 1~2일 | △ LangChain 개념 추가 학습 필요 |
| **커뮤니티 피드백** | 적음 (신생) | 중간 (Python 기반 자료 참고) |

### 평가 기준별 판정

**1. 요구사항 변경 시 유연한가?**

```
Spring AI
  장점: 추상화가 얇아 LLM 호출 흐름을 직접 제어하기 쉬움
        대화 플로우 변경 시 코드가 직관적
  단점: 복잡한 에이전트 패턴(멀티스텝 추론)은 직접 구현해야 함
  → 이 프로젝트 수준(5턴 대화)에서는 충분

LangChain4j
  장점: Chain, Agent, ConversationalMemory 등 패턴이 이미 구현되어 있음
        복잡한 멀티스텝 플로우 변경이 상대적으로 쉬움
  단점: 추상화 레이어가 두꺼워 내부 동작 파악이 어려움
  → 이 프로젝트보다 복잡한 에이전트에 유리
```

**2. 단기간에 습득 가능한가?**

```
Spring AI  ✅ 우위
  - 이미 Spring Boot 4.x, application.yml, @Component, @Service 익숙
  - spring.ai.anthropic.api-key=xxx 설정 한 줄로 Claude 연동
  - 기존 프로젝트에 의존성 추가만으로 바로 시작 가능
  - Micrometer, Actuator 이미 프로젝트에 있어 추가 설정 없음

LangChain4j  △
  - LangChain 고유 개념(Chain, Memory, Retriever) 별도 학습 필요
  - Spring과 통합하려면 설정 코드 직접 작성
  - 한국어 예제가 적어 트러블슈팅 비용 높음
```

**3. 로깅/추적이 가능한가?**

```
Spring AI  ✅ 이미 갖춰진 인프라 재사용
  - Micrometer → Prometheus → Grafana (이미 프로젝트에 있음)
  - LLM 호출 횟수, 응답 시간 메트릭 바로 수집 가능
  - 토큰 사용량 로깅 → API 비용 추적

LangChain4j  △
  - LangSmith 연동 시 RAG 추적 가시성 높음 (어떤 문서가 검색됐는지)
  - 단, LangSmith는 별도 계정/설정 필요 (포트폴리오 수준에서 오버스펙)
  - Grafana 연동은 직접 구현 필요
```

### 결정

```
✅ Spring AI 선택

이유:
  1. 단기 습득 (1순위): Spring Boot 기존 지식 그대로 활용
     → application.yml 설정, @Service, @Component 패턴 동일
  2. 기존 인프라 재사용: Micrometer/Prometheus/Grafana 이미 있음
     → LLM 응답 시간, 토큰 비용 추적을 추가 설정 없이 바로 시작
  3. 이 프로젝트 복잡도에 적합: 5턴 대화, RAG + Function Calling
     → LangChain4j의 고급 에이전트 패턴까지 필요하지 않음

포기하는 것:
  - LangChain4j의 풍부한 체인/에이전트 패턴
  - LangSmith 수준의 RAG 추적 가시성
  → 단, RAG 검색 결과를 로그로 직접 출력하면 포트폴리오 수준에서 충분

면접 어필:
  "Spring AI를 선택한 이유는 기존 Micrometer 인프라를
   LLM 비용/응답 추적에 재사용하기 위해서입니다.
   LangChain4j의 에이전트 추상화가 풍부하지만,
   이 프로젝트의 대화 복잡도에는 오버스펙이라 판단했습니다."
```

### Spring AI 핵심 컴포넌트 (이 프로젝트 사용 범위)

```java
// 1. LLM 호출 (Prompt Engineering + 다중 턴)
ChatClient chatClient;  // Claude API 래핑

// 2. Function Calling
@Tool  // getRecentOrder, getProductUnit, getRecentDeliveryAddress

// 3. RAG
VectorStore vectorStore;     // ES kNN 연결
VectorStoreRetriever retriever;
QuestionAnswerAdvisor advisor;  // 검색 결과 자동 프롬프트 주입

// 4. 스트리밍
Flux<String> stream = chatClient.stream(...);  // 3초 응답 목표
```

### 1. 비즈니스 목표

#### 해결하려는 문제

```
기존 주문 방식의 불편함:
  - 앱에서 카테고리 탐색 → 상품 검색 → 장바구니 → 결제 정보 입력
  - 평균 5~10분 소요 (화면 여러 번 이동)
  - 원하는 상품 이름을 정확히 알아야 검색 가능
  - 타이핑이 불편한 환경 (이동 중, 주방 등)

목표:
  말 한마디로 주문 완료 — 마트 직원에게 말하듯 자연스럽게
```

#### 측정 가능한 목표

| 항목 | 현재 (기존 앱) | 목표 |
|------|--------------|------|
| 주문 완료까지 소요 시간 | 5~10분 | **1분 이내** |
| 주문 완료까지 대화 턴 수 | - | **평균 5턴 이내** |
| 의도 인식 성공률 | - | **일상 주문 표현 90% 이상** |
| 포트폴리오 어필 포인트 | CQRS, Outbox | RAG + Function Calling + 음성 주문 추가 |

### 2. 사용자 니즈

#### 핵심 불편함 (Why 음성/챗봇인가)

```
"뭘 살지는 알겠는데, 앱에서 찾기가 귀찮다"

"주문할 때 손이 자유롭지 않다"
  → 주방에서 요리 중, 이동 중 음성으로 주문하고 싶음

"이전 주문을 매번 다시 입력하기 번거롭다"
  → "저번이랑 똑같이 해줘" 한마디로 재주문하고 싶음
```

#### 사용자 기대 경험 (점원처럼)

```
좋은 점원의 특징:
  ✅ 말이 끝나자마자 바로 반응 (기다리게 하지 않음)

  ✅ 단골이면 이전 구매 기억함 ("저번이랑 똑같이요")
  ✅ 결제/배송 확인을 자연스럽게 진행
  ❌ 모호한 말에 "이해할 수 없습니다" 로 끊지 않음
```

### 3. 기능 요구사항

#### 필수 기능 (Phase 1~2 목표)

**F1. 음성/텍스트 입력 전환**
- 마이크 버튼으로 음성 입력 (Web Speech API, 한국어)
- 음성 인식 실패 시 텍스트 입력으로 자동 폴백
- 인식된 텍스트를 화면에 표시 (사용자 확인용)

**F2. 자연어 상품 추천**
- "달달한 간식", "제주산 과일", "아이 간식" 같은 자연어 질문 처리
- 우리 마트 상품 기반 Top 3~5 추천 (RAG)
- 재고 없는 상품은 추천에서 제외

**F3. 주문 의도 파악 및 수집**
- 상품명/수량 자연어 추출 ("감귤 세 개" → productName: 감귤, quantity: 3)
- 단위 모호성 자동 해소 ("2개" → 상품 단위 조회 후 "2박스입니까?" 확인)
- 구어체 수량 처리 ("두 배로", "조금만", "한 박스")

**F4. 이전 주문/배송지 참조**
- "저번이랑 똑같이요" → 최근 주문 이력 조회 후 반영
- "지난번 배송지로요" → 이전 배송지 자동 불러오기
- 참조한 내용을 대화에서 명시 ("지난번 2박스 기준으로 진행할게요")

**F5. 결제/배송지 수집**
- 결제 수단 선택 유도 (카드 / 현금 선불)
- 배송지 확인 또는 신규 입력
- 최종 주문 내용 요약 후 확인 받기

**F6. 주문 생성 연결**
- 확인 완료 시 기존 POST /api/orders 호출 (order-service 재사용)
- 주문 성공 시 tossOrderId 안내

#### 선택 기능 (Phase 4, 포트폴리오 완성도용)

**F7. TTS 응답** — AI 답변을 음성으로 읽어줌 (Web Speech API synthesis)
**F8. 대화 이력 표시** — 채팅 UI처럼 주고받은 내용 화면 표시

### Step 1. 데이터 로딩 — 소스 연결과 수집

#### 1. 어떤 데이터를 연결할 것인가?

**RAG 대상 데이터 = 상품 데이터만**

```
왜 상품 데이터만인가?

주문 이력, 배송지  → Function Calling (정확한 단건 조회)
                    → 벡터 인덱싱 불필요

상품 데이터        → RAG (자연어로 유사 상품 검색)
                    → "달달한 간식" → 의미적으로 가까운 상품 찾기
                    → 벡터 인덱싱 필요
```

| 데이터 | 소스 | 연결 방식 | 인덱싱 목적 |
|--------|------|-----------|------------|
| 상품 정보 | product-service MySQL | Kafka `product.registered.v1` 수신 (이미 설계됨) | 자연어 상품 추천 |
| 카테고리 정보 | product-service MySQL | 상품 데이터에 포함 (JOIN) | "과일", "간식" 카테고리 검색 |

> **이미지, 상품 상세 HTML은 제외**: 텍스트 데이터만 먼저.
> 이미지 검색(멀티모달)은 이후 별도 파이프라인으로 추가 가능.

**초기 전체 색인 대상 (서비스 최초 기동 시)**

```
product-service MySQL → products 테이블 전체 조회
→ 예상 데이터 양: 포트폴리오 수준 100~500개 상품
→ 배치 처리로 일괄 임베딩 후 ES 저장
```

**증분 색인 (이후 상품 등록/변경 시)**

```
product.registered.v1 Kafka 이벤트 수신
→ 신규/변경 상품만 임베딩 후 ES upsert
→ 이미 CLAUDE.md에 설계된 이벤트 그대로 활용
```

#### 2. 어떤 포맷을 파싱할 것인가?

**원본 데이터 (MySQL product 테이블)**

```sql
products
  id            BIGINT       -- productId
  name          VARCHAR      -- "제주 감귤"
  description   TEXT         -- "제주도에서 재배한 당도 높은 감귤. 선물용으로 인기."
  price         BIGINT       -- 15000
  category_id   BIGINT
  unit          VARCHAR      -- "박스"
  unit_size     INT          -- 10  (1박스 = 10개)
  stock         INT          -- 재고 수량
  image_url     VARCHAR      -- S3 URL (임베딩 제외)
```

**임베딩용 텍스트 포맷 (파싱 결과)**

```
파싱 규칙:
  - image_url, id, category_id 등 숫자/URL 필드 제외
  - 의미 있는 텍스트 필드만 조합
  - 가격/단위 정보는 추천 품질에 영향 → 포함

최종 텍스트 포맷:
  "상품명: 제주 감귤
   카테고리: 과일
   설명: 제주도에서 재배한 당도 높은 감귤. 선물용으로 인기.
   단위: 박스 (1박스 10개)
   가격: 15,000원"

→ 이 텍스트를 통째로 임베딩 → 벡터 1개 생성
```

> **설명(description) 필드가 없거나 비어 있는 경우**:
> `"상품명: {name}, 카테고리: {category}"` 최소 포맷으로 폴백

#### 3. 어떻게 데이터를 정제할 것인가?

**정제 규칙**

| 항목 | 원본 예시 | 정제 결과 | 이유 |
|------|-----------|-----------|------|
| HTML 태그 제거 | `"<b>신선한</b> 감귤"` | `"신선한 감귤"` | 태그가 임베딩 품질 저하 |
| 특수문자 정리 | `"감귤!!!! 🍊🍊"` | `"감귤"` | 이모지/과도한 특수문자 제거 |
| 가격 단위 통일 | `"15000"` | `"15,000원"` | 자연어 검색에서 "만오천원" 매칭 향상 |
| 재고 없는 상품 | `stock = 0` | **색인 제외** or `"품절"` 태그 | 품절 상품 추천 방지 |
| 설명 null | `description = null` | 상품명 + 카테고리만으로 구성 | NullPointerException 방지 |
| 중복 공백 | `"감귤   박스"` | `"감귤 박스"` | 토큰 낭비 방지 |

**정제 후 ES 저장 스키마**

```json
{
  "productId": 7,
  "productName": "제주 감귤",
  "categoryName": "과일",
  "price": 15000,
  "unit": "박스",
  "unitSize": 10,
  "inStock": true,
  "embeddingText": "상품명: 제주 감귤 카테고리: 과일 설명: 제주도에서 재배한 당도 높은 감귤 단위: 박스 1박스 10개 가격: 15,000원",
  "embedding": [0.023, -0.041, ...]   // dense_vector, 1536차원 (OpenAI)
}
```

  - 실패 시 해당 배치만 재시도 (Spring Retry)

증분 색인 (product.registered.v1 수신 시):
  - 단건 처리, Rate Limit 영향 없음

### Step 2. 청킹 — 검색 가능한 단위로 분할

> **핵심**: 청크가 너무 작으면 맥락 손실, 너무 크면 컨텍스트 낭비 + 속도 저하.
> 데이터 분할 시 의미 손실을 어떻게 보완할지가 핵심.
> 메타데이터는 선택이 아닌 필수 — "어디서 나온 데이터인지"가 검색 정확도를 좌우.

---

#### 1. 어떤 전략으로 분할할 것인가?

**상품 데이터의 특성**

```
일반 문서 (PDF, 기사):  긴 텍스트 → 의미 단위로 잘라야 함 → 청킹 전략 중요
상품 데이터:           각 상품이 이미 독립적인 의미 단위 → 자연 분할 경계 존재
```

**선택: 상품 1개 = 청크 1개 (Natural Unit Chunking)**

```
왜 1상품 1청크인가?

Option A — 상품 전체를 1청크
  "상품명: 제주 감귤 카테고리: 과일 설명: 제주도에서 재배한 당도 높은 감귤.
   선물용으로 인기. 단위: 박스(10개) 가격: 15,000원"
  ✅ 상품명 + 카테고리 + 설명 + 가격이 한 벡터에 응집
  ✅ "달달한 과일 선물용" 검색 시 설명의 맥락이 살아있음
  ✅ 검색 결과 = 상품 1개, 바로 추천으로 연결 가능

Option B — 상품명/설명 분리 청킹
  청크1: "제주 감귤, 과일, 15,000원"
  청크2: "제주도에서 재배한 당도 높은 감귤. 선물용으로 인기."
  ❌ 검색 후 두 청크를 다시 합쳐야 함 (복잡도 증가)
  ❌ 청크2 단독으로는 어떤 상품인지 알 수 없음 (맥락 손실)
  ❌ 포트폴리오 수준 상품 수에서 불필요한 복잡도

Option C — 문장 단위 청킹 (Sentence Chunking)
  ❌ "제주도에서" 라는 단어 하나가 벡터가 됨
  ❌ 가격/단위 정보 소실 → 추천 품질 저하
  ❌ 이 프로젝트 데이터 특성에 부적합
```

**결론: Option A (1상품 1청크)**

---

#### 2. 크기와 중복은 얼마로 할 것인가?

**청크 크기 (토큰 기준)**

```
상품 데이터 1개 예상 토큰:
  상품명:    5~15 토큰   ("제주 감귤", "유기농 꿀사과 2kg")
  카테고리:  3~8 토큰    ("과일", "아이 간식")
  설명:      30~150 토큰 (없으면 0)
  가격/단위: 10~20 토큰
  ─────────────────────
  합계:      50~200 토큰 (평균 약 100토큰)

OpenAI text-embedding-3-small 최대 입력: 8,191 토큰
→ 상품 1개 100토큰 << 8,191 토큰 한계 → 청크 분할 자체가 불필요
→ 설명이 아무리 길어도 500토큰 이하면 그대로 1청크 처리
```

**오버랩(Overlap)**

```
불필요
→ 오버랩은 순차 문서(PDF, 기사)에서 청크 경계의 맥락 손실을 보완하는 기법
→ 상품 데이터는 각 청크가 독립적, 이전/다음 상품과 관계 없음
```

**설명이 500토큰을 초과하는 경우 (예외 처리)**

```
긴 설명 (500토큰 초과) → 앞 300토큰만 사용 + "..." 처리
이유: 상품 핵심 정보(이름, 카테고리, 가격)는 앞부분에 집중됨
     뒷부분은 상세 스펙이라 추천 검색에서 영향 미미
```

#### 3. 어떤 메타데이터를 함께 저장할 것인가?

> **핵심**: 메타데이터는 벡터 검색 전 **사전 필터링**에 사용.
> "재고 있는 과일 중에서 달달한 것" = `inStock: true AND category: 과일` 필터 후 kNN 검색
> → 필터가 정확할수록 관련 없는 상품이 추천되는 오류 감소

**저장할 메타데이터와 역할**

| 필드 | 타입 | 역할 | 없으면 |
|------|------|------|--------|
| `productId` | Long | 검색 후 product-service API 조회용 키 | 조회 불가 |
| `productName` | String | 추천 응답에 바로 표시 | API 재조회 필요 |
| `categoryName` | String | 사전 필터 ("과일만 보여줘") | 카테고리 필터 불가 |
| `price` | Long | 가격 범위 필터 ("만원 이하") | 가격 필터 불가 |
| `inStock` | Boolean | **가장 중요** — 품절 상품 추천 차단 | 품절 상품이 추천됨 |
| `unit` | String | 주문 시 단위 모호성 해소 ("박스") | Function Calling 추가 호출 필요 |
| `unitSize` | Integer | "1박스 = 10개" 안내용 | 단위 설명 불가 |
| `updatedAt` | DateTime | 오래된 색인 감지, 재색인 판단 | 데이터 신선도 파악 불가 |

**ES 최종 저장 구조**

```json
{
  "productId": 7,
  "productName": "제주 감귤",
  "categoryName": "과일",
  "price": 15000,
  "inStock": true,
  "unit": "박스",
  "unitSize": 10,
  "updatedAt": "2026-04-09T10:00:00",
  "embeddingText": "상품명: 제주 감귤 카테고리: 과일 설명: 제주도에서 재배한 당도 높은 감귤 선물용으로 인기 단위: 박스 1박스 10개 가격: 15,000원",
  "embedding": [0.023, -0.041, ...]
}
```

**사전 필터 + kNN 검색 예시 (ES 쿼리)**

```json
{
  "knn": {
    "field": "embedding",
    "query_vector": [...],
    "k": 5,
    "num_candidates": 50,
    "filter": {
      "bool": {
        "must": [
          { "term": { "inStock": true } }
        ],
        "should": [
          { "term": { "categoryName": "과일" } }
        ]
      }
    }
  }
}
```

> `inStock: true` 필터는 항상 적용 (품절 상품 추천 방지).
> `categoryName` 필터는 소비자 발화에서 카테고리가 명확할 때만 적용.

### 청킹 전략 요약

```
전략:    1상품 1청크 (Natural Unit)
크기:    평균 100토큰, 최대 500토큰 제한
오버랩:  없음 (상품 데이터는 독립 단위)
메타데이터: productId, productName, categoryName, price, inStock, unit, unitSize
사전 필터: inStock: true 항상 + categoryName 조건부
```

### Step 3. 임베딩 — 텍스트를 벡터로 변환

> **유의사항**:
> - 파운데이션 모델로 먼저 검증. 로딩/청킹이 검증되지 않은 상태에서 파인튜닝은 오버킬
> - 차원 수는 높을수록 정확도 향상이지만 비용/속도도 함께 증가
> - 파인튜닝은 나중에 품질 문제가 생겼을 때 검토

---

#### 1. 어떤 임베딩 모델을 선택할 것인가?

**후보 비교**

| 모델 | 제공사 | 한국어 품질 | 차원 | 비용 | 운영 방식 | Spring AI 지원 |
|------|--------|------------|------|------|-----------|---------------|
| `text-embedding-3-small` | OpenAI | ⭐⭐⭐⭐ | 1536 | $0.02/1M 토큰 | API 호출 | ✅ |
| `text-embedding-3-large` | OpenAI | ⭐⭐⭐⭐⭐ | 3072 | $0.13/1M 토큰 | API 호출 | ✅ |
| `text-embedding-ada-002` | OpenAI | ⭐⭐⭐⭐ | 1536 | $0.10/1M 토큰 | API 호출 | ✅ (구버전) |
| `ko-sroberta-multitask` | HuggingFace | ⭐⭐⭐⭐⭐ | 768 | 무료 | 로컬 서버 | △ 직접 연동 |
| Upstage Embeddings | Upstage | ⭐⭐⭐⭐⭐ | 4096 | 유료 | API 호출 | △ 직접 연동 |

**이 프로젝트 상황 분석**

```
한국어 상품명/설명 임베딩이 목적
  → "감귤", "달달한 간식", "아이 간식" 같은 일상 표현
  → 전문 의학/법률 용어 아님 → 일반 다국어 모델로 충분

포트폴리오 수준, 상품 500개 이하
  → 초기 색인 500개 × 100토큰 = 50,000 토큰
  → text-embedding-3-small 기준 $0.001 (1달러 미만)

Spring AI 기본 지원 필요
  → 설정 1줄로 교체 가능한 구조가 중요
  → 로컬 HuggingFace 모델은 별도 서버 운영 필요 → 포트폴리오에 불필요한 복잡도
```

**결정: `text-embedding-3-small`**

```
선택 이유:
  1. 파운데이션 모델 검증 원칙 — 가장 검증된 모델로 시작
  2. Spring AI 기본 지원 — application.yml 설정 1줄
  3. 한국어 일상 표현 충분히 커버
  4. 비용 무시 가능한 수준 (포트폴리오)
  5. 나중에 모델 교체 시 Spring AI EmbeddingModel 인터페이스만 교체

Spring AI 설정:
  spring.ai.openai.embedding.options.model=text-embedding-3-small
```

**모델 교체 전략 (나중에 품질 문제 시)**

```
1단계: text-embedding-3-small (지금)
2단계: text-embedding-3-large (정확도 부족 시)
3단계: Upstage Embeddings (한국어 전문성 필요 시)
→ Spring AI EmbeddingModel 인터페이스 덕분에 코드 변경 없이 교체 가능
```

#### 2. 차원의 크기를 어떻게 설정할 것인가?

**차원 수와 트레이드오프**

```
차원 수 높음 → 의미 표현 정밀도 향상, ES 저장 공간 증가, kNN 검색 속도 감소
차원 수 낮음 → 검색 빠름, 저장 공간 절약, 미세한 의미 차이 손실
```

**이 프로젝트에서의 계산**

```
text-embedding-3-small 기본 차원: 1,536

ES 저장 공간:
  상품 1개 × 1,536차원 × 4바이트(float) = 6KB
  상품 500개 × 6KB = 3MB
  → 저장 공간 문제 없음

kNN 검색 속도:
  500개 상품 × 1,536차원 = 극히 소량
  → 수백만 건도 아니고 수백 건 → 차원 수가 속도에 영향 없음
```

**결정: 기본값 1,536차원 그대로 사용**

```
이유:
  - 상품 500개 수준에서 차원 최적화는 불필요
  - 기본값이 검증된 설정 → 파운데이션 검증 원칙에 부합
  - Matryoshka 차원 축소(256/512/1024)는 수백만 건 이상일 때 의미 있음

text-embedding-3-small의 Matryoshka 지원:
  dimensions: 256, 512, 1024, 1536 중 선택 가능
  → 나중에 상품 규모가 커질 때 256~512로 줄여 비용/속도 최적화 검토
```

#### 3. 파인튜닝이 필요한가?

**판단 기준**

```
파인튜닝이 필요한 경우:
  - 기반 모델이 모르는 전문 용어가 많음 (의료, 법률, 특수 산업)
  - 일반 모델로 검색했을 때 품질이 명확히 낮음 (검증 후 판단)
  - 충분한 학습 데이터가 있음 (최소 수천 건의 쿼리-상품 쌍)

파인튜닝이 불필요한 경우:
  - 일상 언어로 표현 가능한 도메인
  - 로딩/청킹 파이프라인 검증이 아직 안 됨
  - 데이터가 적음 (포트폴리오 수준)
```

**이 프로젝트 판단**

```
❌ 파인튜닝 불필요 (지금은)

근거:
  1. 파이프라인 미검증
     로딩 → 청킹 → 임베딩 → ES 저장 전체 흐름이 아직 구현 전
     "달달한 간식" 검색이 실제로 잘 되는지 검증 전에 파인튜닝은 배보다 배꼽

  2. 도메인 특성
     "감귤", "꿀사과", "아이 간식", "달달한" 등
     → 일반 한국어 일상 표현, 전문 용어 없음
     → text-embedding-3-small이 이미 충분히 학습된 영역

  3. 데이터 부족
     파인튜닝에 필요한 최소 데이터:
       (쿼리, 관련 상품, 비관련 상품) 쌍 수천 건
     포트폴리오 상품 500개로는 학습 데이터 생성 자체가 불가

파인튜닝 검토 시점:
  → Step 4(저장) + Step 5(검색) 구현 후 실제 검색 품질 테스트
  → "달달한 간식" 검색 결과가 기대와 많이 다를 때
  → 그때도 파인튜닝 전에 프롬프트/청킹 개선을 먼저 시도
```

#### 임베딩 파이프라인 코드 구조 (Spring AI)

```java
// application.yml
spring:
  ai:
    openai:
      api-key: ${OPENAI_API_KEY}
      embedding:
        options:
          model: text-embedding-3-small
          # dimensions: 1536  기본값, 명시 불필요

// ProductEmbeddingService.java
@Service
@RequiredArgsConstructor
public class ProductEmbeddingService {

    private final EmbeddingModel embeddingModel;  // Spring AI 인터페이스
    private final VectorStore vectorStore;         // ES 연결

    // 상품 1개 임베딩 + 저장
    public void indexProduct(ProductDocument product) {
        String text = buildEmbeddingText(product);   // Step 1 정제 텍스트
        Document doc = new Document(text, product.toMetadata());
        vectorStore.add(List.of(doc));               // 임베딩 + ES 저장 자동 처리
    }

    // 전체 초기 색인 (배치 100개씩)
    public void indexAll(List<ProductDocument> products) {
        Lists.partition(products, 100).forEach(batch -> {
            List<Document> docs = batch.stream()
                .map(p -> new Document(buildEmbeddingText(p), p.toMetadata()))
                .toList();
            vectorStore.add(docs);
            // Rate Limit 여유: 100개 × 100토큰 = 10,000토큰, 100ms 딜레이
        });
    }

    private String buildEmbeddingText(ProductDocument p) {
        return "상품명: %s 카테고리: %s 설명: %s 단위: %s(%d개) 가격: %,d원"
            .formatted(p.name(), p.category(), p.description(),
                       p.unit(), p.unitSize(), p.price());
    }
}
```

### Step 4. 저장소 — 벡터 데이터 저장과 검색

> **유의사항**:
> - Managed vs Self-hosted 결정 필요. Docker로 먼저 테스트 권장
> - **데이터 삭제 기준 설정 필수** — 없으면 삭제된 상품이 계속 검색에 노출됨

---

#### 1. 어떤 저장소를 선택할 것인가?

**후보 비교**

| 저장소 | 방식 | 벡터 검색 | Spring AI 지원 | 운영 복잡도 | 이 프로젝트 적합성 |
|--------|------|-----------|---------------|------------|-----------------|
| **Elasticsearch** | Self-hosted | ✅ kNN (8.x~) | ✅ ElasticsearchVectorStore | 중간 | ✅ **이미 계획됨** |
| Pinecone | Managed SaaS | ✅ 전용 벡터 DB | ✅ | 낮음 | △ 신규 외부 의존성 |
| Weaviate | Self-hosted | ✅ 전용 벡터 DB | ✅ | 중간 | △ 신규 인프라 필요 |
| Redis Vector | Self-hosted | ✅ Redis Stack | △ | 낮음 | ❌ Redis Sentinel과 다른 제품 |
| pgvector | Self-hosted | ✅ PG 확장 | ✅ | 낮음 | ❌ 이 프로젝트 MySQL 사용 |

**결정: Elasticsearch**

```
이유:
  1. 이미 search-service(8090) 계획에 포함 — 신규 인프라 없음
  2. Docker compose.yaml에 추가만 하면 로컬 즉시 사용 가능
  3. Spring AI ElasticsearchVectorStore 기본 지원
  4. 키워드 검색 + 벡터 검색 동시 가능 (Hybrid Search)
     → "감귤 15000원 이하" = 가격 필터(키워드) + 의미 검색(벡터) 조합
  5. 이미 Micrometer로 모니터링 인프라 갖춰져 있음

Pinecone을 선택하지 않은 이유:
  - 신규 외부 SaaS 의존성 추가 = 장애 포인트 증가
  - 이미 ES가 있는데 별도 벡터 DB는 중복 인프라
  - 포트폴리오에서 "왜 두 개를 쓰나요?" 설명이 어색함
```

**로컬 vs 운영 환경 구성**

```
로컬 (Docker):
  elasticsearch:8.13.0 단일 노드
  compose.yaml에 추가
  xpack.security.enabled=false (개발 편의)
  포트: 9200

운영 (EC2, 현재는 미적용 — 포트폴리오 수준):
  단일 노드로 시작, 필요 시 3노드 클러스터 확장
  AWS OpenSearch Service 전환도 고려 가능 (관리형)
```

  },
  "mappings": {

#### 데이터 삭제 기준 (필수)

> 삭제 기준이 없으면 판매 종료된 상품, 가격이 바뀐 상품이 계속 추천에 노출됨

**이벤트별 삭제/갱신 전략**

| 상황 | ES 처리 | 임베딩 재계산 | 이유 |
|------|---------|-------------|------|
| 상품 신규 등록 | insert | ✅ 필요 | 새 벡터 생성 |
| 상품명/설명 변경 | upsert (productId 기준) | ✅ 필요 | 텍스트 변경 = 벡터 변경 |
| 가격 변경 | partial update (`price` 필드만) | ❌ 불필요 | 가격은 메타데이터, 벡터 영향 없음 |
| 재고 변경 | partial update (`inStock` 필드만) | ❌ 불필요 | 메타데이터 갱신만 |
| 상품 삭제 | **delete (productId 기준)** | ❌ 불필요 | 즉시 검색 대상에서 제거 |
| 임베딩 모델 변경 | **전체 삭제 후 재색인** | ✅ 전체 필요 | 모델 다르면 벡터 비교 불가 |

**상품 삭제 이벤트 처리 (중요)**

```
product-service에서 상품 삭제 시:
  → product.deleted.v1 Kafka 이벤트 발행 (현재 미설계 → 추가 필요)
  → chat-service에서 수신 → ES에서 productId로 즉시 삭제

이벤트 없이 삭제가 발생하는 경우 (안전망):
  → 주기적 동기화 Job: product-service 전체 목록 vs ES 목록 비교
  → ES에만 있고 MySQL에 없는 항목 삭제 (야간 배치)
```

**전체 재색인 전략**

```
언제 필요한가:
  - 임베딩 모델 교체 시 (text-embedding-3-small → large)
  - embeddingText 포맷 변경 시 (청킹 전략 수정)
  - ES 인덱스 매핑 변경 시

방법:
  1. 신규 인덱스 생성 (products_v2)
  2. 전체 상품 재색인 (products_v2에 저장)
  3. ES alias 교체 (products → products_v2)
  4. 구버전 인덱스 삭제 (products_v1)
  → 서비스 중단 없이 재색인 가능 (Blue-Green 인덱싱)
```

### Step 5. 검색 — 관련성 높은 청크 추출

> **유의사항**:
> - 메타데이터 필터링으로 검색 효율 향상 (카테고리, 재고 상태)
> - 청크 개수 제한 필수 — 너무 많으면 컨텍스트 초과
> - 유사도 기준: 너무 낮으면 관련 없는 결과, 너무 높으면 결과 없음
> - 검색 정밀도는 평가 단계에서 반드시 검토

---

#### 1. 어떤 검색 전략을 사용할 것인가?

**검색 전략 비교**

| 전략 | 방식 | 잘 맞는 쿼리 | 못 잡는 쿼리 |
|------|------|------------|------------|
| **Dense (벡터)** | 의미 유사도 (코사인) | "달달한 간식", "아이 선물용" | "감귤 정확히" (오히려 유사 상품도 포함) |
| **Sparse (키워드)** | BM25 + nori 형태소 | "제주 감귤", 정확한 상품명 | "달콤한 것" (의미 검색 불가) |
| **Hybrid** | Dense + Sparse 결합 | 두 경우 모두 | 없음 |

**이 프로젝트 쿼리 유형 분석**

```
"달달한 간식 추천해줘"    → 의미 기반 → Dense 유리
"감귤 있어요?"           → 키워드 기반 → Sparse 유리
"만원 이하 과일 추천"    → 의미 + 가격 필터 → Hybrid + 메타데이터 필터
"제주산 달콤한 과일"     → 키워드 + 의미 혼합 → Hybrid 유리
```

**결정: Hybrid Search (Dense + Sparse + 메타데이터 사전 필터)**

```
ES 8.x RRF (Reciprocal Rank Fusion):
  - kNN 벡터 검색 결과 순위
  - BM25 키워드 검색 결과 순위
  → 두 순위를 RRF로 합산 → 최종 Top-K 반환
  → ES에서 단일 쿼리로 처리 가능
```

**ES Hybrid Search 쿼리 구조**

```json
{
  "retriever": {
    "rrf": {
      "retrievers": [
        {
          "knn": {
            "field": "embedding",
            "query_vector": [...],
            "k": 10,
            "num_candidates": 30,
            "filter": { "term": { "inStock": true } }
          }
        },
        {
          "standard": {
            "query": {
              "bool": {
                "must": { "match": { "productName": "감귤" } },
                "filter": { "term": { "inStock": true } }
              }
            }
          }
        }
      ],
      "rank_window_size": 10,
      "rank_constant": 60
    }
  },
  "size": 5
}
```

> **항상 적용하는 사전 필터**: `inStock: true`
> 카테고리 필터는 쿼리에서 카테고리가 명확할 때만 추가

**검색할 청크 개수 (Top-K) 결정**

```
ES에서 가져오는 수: 10 (num_candidates: 30)
LLM에 전달하는 수: 3~5

왜 3~5개인가:
  상품 1개 = 약 100토큰
  5개 × 100토큰 = 500토큰 (전체 컨텍스트의 일부, 안전)
  10개 × 100토큰 = 1,000토큰 (가능하지만 불필요한 상품 포함 위험)
  20개 이상 → LLM이 관련 없는 상품도 추천하기 시작함

단계별 필터링:
  ES kNN 후보: 30개 (num_candidates)
  → RRF 결합 후: 10개
  → 유사도 임계값 필터: 5개 이하
  → LLM에 전달: 최대 5개
```

**유사도 임계값 (Score Threshold)**

```
코사인 유사도 범위: 0.0 ~ 1.0

초기 설정: 0.6 (보수적)
  0.6 미만 → 관련성 낮음 → LLM에 전달하지 않음
  결과가 0건이면 → "해당 상품을 찾기 어렵습니다. 다르게 말씀해 주실 수 있나요?"

실제 의미:
  1.0: 동일한 텍스트
  0.9: 거의 같은 의미 ("감귤" vs "귤")
  0.7: 관련 있음 ("달달한 과일" vs "감귤")
  0.5: 약한 연관 (일반적으로 노이즈)
  0.3 이하: 무관

운영 후 튜닝 방향:
  결과가 너무 없음 → 0.5로 낮춤
  관련 없는 상품이 자주 추천됨 → 0.7로 높임
```

#### 2. 사용자 질문을 최적화할 것인가?

**쿼리 최적화 기법 비교**

| 기법 | 방법 | 효과 | 이 프로젝트 적용 |
|------|------|------|---------------|
| **직접 임베딩** | 질문 그대로 벡터화 | 기본, 가장 단순 | ✅ Phase 1 |
| **메타데이터 추출** | 질문에서 카테고리/가격 추출 → 사전 필터 적용 | 검색 범위 축소, 정확도 향상 | ✅ Phase 1 |
| **HyDE** | 가상 상품 설명 생성 후 임베딩 | 질문-문서 벡터 공간 불일치 해소 | 🔜 Phase 2 (품질 부족 시) |
| **Multi-Query** | 질문 여러 버전 생성 후 각각 검색 | 검색 범위 확장 | 🔜 Phase 2 (필요 시) |
| **Query Expansion** | 동의어 추가 ("달달한" → "달콤한, 단맛") | Sparse 검색 개선 | 🔜 Phase 2 |

**Phase 1에서 적용할 최적화: 메타데이터 추출**

```
사용자 질문에서 카테고리와 가격 범위를 추출해 사전 필터로 활용
→ 벡터 검색 전에 후보 축소 → 정확도 향상, 속도 개선

예시:
  "만원 이하 과일 추천해줘"
    → price ≤ 10000 (가격 필터)
    → categoryName: "과일" (카테고리 필터)
    → 필터 후 남은 상품에서 kNN 벡터 검색

추출 방법: LLM에게 먼저 구조화 요청
  System: "질문에서 카테고리와 가격 조건을 JSON으로 추출하세요"
  User: "만원 이하 과일 추천해줘"
  → { "category": "과일", "maxPrice": 10000 }
  → 이 필터를 ES 쿼리에 추가
```

**HyDE를 나중으로 미루는 이유**

```
HyDE (Hypothetical Document Embedding):
  "달달한 간식" → LLM이 가상 상품 설명 생성
  → "당도 높은 제철 과일, 아이들이 좋아하는 달콤한 맛..."
  → 이 가상 설명을 임베딩 → 벡터 검색

효과: 질문(짧음)과 문서(긺) 사이의 임베딩 공간 불일치 해소
단점: LLM 호출이 검색 전에 1회 추가 → 응답 시간 증가, 비용 증가

판단:
  - 파운데이션 모델로 직접 임베딩 먼저 검증
  - 검색 품질이 부족하다고 판단될 때 도입
  - 3초 응답 목표에서 LLM 추가 호출은 부담
```

---

### 검색 단계 요약

```
전략:    Hybrid Search (kNN + BM25 + RRF)
사전필터: inStock: true 항상 + 카테고리/가격 조건부
Top-K:   ES 후보 30개 → RRF 10개 → 임계값 필터 → LLM 전달 최대 5개
임계값:  코사인 유사도 0.6 (운영 후 0.5~0.7 범위에서 튜닝)
질문최적화:
  Phase 1: 질문에서 카테고리/가격 추출 → 사전 필터 (단순, 빠름)
  Phase 2: HyDE, Multi-Query (품질 부족 시)
```

#### 1. 페르소나를 어떻게 정의할 것인가?

**페르소나 설계 원칙**

```
목표: "마트 직원처럼" — 친근하고 빠르고 정확하게

좋은 마트 직원의 특징:
  ✅ 짧고 자연스러운 말투 ("감귤 있어요! 2박스 하실 건가요?")
  ✅ 모르면 솔직하게 ("죄송해요, 해당 상품은 현재 없어요")
  ✅ 단골 기억 ("저번에 감귤 2박스 주문하셨죠?")
  ✅ 주문 완결에 집중 (잡담 최소화)
  ❌ "저는 AI입니다" 반복 강조 안 함
  ❌ 없는 상품을 있는 척하지 않음
  ❌ 너무 긴 설명 (점원이 3분 설명하면 손님이 지침)
```

**시스템 프롬프트 — 페르소나 정의**

```
당신은 allmart 마트의 주문 도우미입니다.
소비자가 상품을 찾고 주문하는 전 과정을 도와주세요.

말투:
- 짧고 자연스러운 한국어 (1~3문장)
- 친근하되 너무 격식 없지 않게
- "~요", "~까요?" 어미 사용

반드시 지킬 규칙:
- 아래 [상품 목록]에 있는 상품만 추천하세요
- 목록에 없는 상품은 "현재 취급하지 않는 상품이에요"라고 안내하세요
- 재고가 없는 상품은 추천하지 마세요
- 가격, 단위는 [상품 목록]의 정보를 그대로 사용하세요
- 주문 확정 전 반드시 상품명, 수량, 금액을 한번 확인해 주세요
```

#### 2. 컨텍스트 제약을 어떻게 설정할 것인가?

**컨텍스트 창 예산 (Claude Sonnet 기준 200K 토큰)**

```
구성 요소              예상 토큰    비고
──────────────────────────────────────
시스템 프롬프트          200토큰    페르소나 + 규칙
검색된 상품 목록 (5개)   500토큰    1개당 약 100토큰
대화 이력 (최근 10턴)    800토큰    1턴당 약 80토큰
현재 사용자 질문          50토큰
──────────────────────────────────────
합계                   1,550토큰   200K 한계 대비 1% 미만
```

> 포트폴리오 수준에서는 컨텍스트 초과 위험 없음.
> 단, 대화 이력을 무제한 누적하면 비용 증가 → 최근 10턴으로 제한.

**컨텍스트 제약 규칙**

```
대화 이력: 최근 10턴만 유지 (오래된 턴 앞에서 제거)
상품 목록: 검색 결과 최대 5개 (Step 5에서 결정)
응답 길이: 최대 150토큰 (3문장 이내 — 점원은 길게 말하지 않음)
```

**상품 목록 컨텍스트 주입 포맷**

```
[상품 목록]
1. 제주 감귤 | 카테고리: 과일 | 가격: 15,000원 | 단위: 박스(10개) | 재고: 있음
   설명: 제주도에서 재배한 당도 높은 감귤. 선물용으로 인기.

2. 꿀사과 | 카테고리: 과일 | 가격: 12,000원 | 단위: 박스(5개) | 재고: 있음
   설명: 달콤하고 아삭한 사과. 아이 간식으로 추천.

3. 딸기 | 카테고리: 과일 | 가격: 18,000원 | 단위: 팩(500g) | 재고: 없음
   → [재고 없음 → LLM에 전달하지 않음]
```

> 재고 없는 상품은 프롬프트에 아예 포함하지 않음.
> `inStock: false` 상품은 ES 검색 단계에서 이미 필터링됨.

#### 3. 어떤 프롬프팅 기법을 활용할 것인가?

**기법별 적용**

**① Role Prompting (페르소나)**
```
"당신은 allmart 마트의 주문 도우미입니다"
→ LLM이 일관된 말투와 역할 유지
→ 환각 감소 (역할 범위 이탈 방지)
```

**② Grounding (근거 기반 응답)**
```
"아래 [상품 목록]에 있는 상품만 추천하세요"
→ 환각 방지의 핵심 — LLM이 모르는 상품을 만들어내는 것 차단
→ 목록에 없으면 솔직하게 "없다"고 말하도록 강제

나쁜 예: 프롬프트에 제약 없음
  → "한라봉도 맛있어요" (목록에 없는 상품 추천)

좋은 예: Grounding 적용
  → "현재 취급하지 않는 상품이에요. 감귤은 어떠세요?"
```

**③ Few-Shot (대화 예시 제공)**
```
# 나쁜 대화 예시 포함 안 함 — 좋은 예시만

예시 대화:
User: 달달한 간식 추천해줘
Assistant: 꿀사과(12,000원)랑 제주 감귤(15,000원) 있어요.
           아이 간식으로는 꿀사과가 많이 팔려요. 어떤 걸로 할까요?

User: 감귤로 할게요
Assistant: 감귤은 1박스(10개) 단위예요. 몇 박스 드릴까요?

→ LLM이 응답 길이, 말투, 질문 방식을 학습
→ 시스템 프롬프트에 2~3개 예시면 충분 (더 많으면 토큰 낭비)
```

**④ Output Format Control (구조화 출력)**
```
대화 응답은 자연어지만, 주문 확정 단계에서는 구조화 필요

최종 주문 확인 단계:
  System: "주문 확정 시 아래 JSON 형식으로 함께 출력하세요
           (사용자에게 보이는 텍스트와 별도로)
           ##ORDER_CONFIRM##
           { 'productId': ..., 'quantity': ..., 'paymentMethod': ... }
           ##END##"

→ 백엔드에서 ##ORDER_CONFIRM## 파싱 → POST /api/orders 호출
→ 자연어 응답과 구조화 데이터를 한 번에 처리
```

### Step 7. 생성 — 최종 답변 생성

> **유의사항**:
> - 파운데이션 모델로 먼저 검증. 색인/검색/증강이 제대로 작동하는지 확인이 우선
> - 작은 작업에 큰 모델 → 속도/비용 낭비. 큰 데이터에 작은 모델 → 답변 불가

#### 1. 오픈소스 vs 독점 모델

**비교표**

| | **독점 (Claude / GPT)** | **오픈소스 (Llama / Mistral)** |
|---|---|---|
| **한국어 품질** | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ (모델마다 다름) |
| **API 편의성** | ✅ 바로 호출 | ❌ 서버 직접 운영 필요 |
| **비용** | 사용량 과금 | 서버 비용 (고정) |
| **응답 속도** | 빠름 (인프라 최적화됨) | 서버 성능에 의존 |
| **Spring AI 연동** | ✅ 기본 지원 | △ Ollama 경유 가능 |
| **데이터 프라이버시** | 외부 전송 | ✅ 로컬 처리 |
| **포트폴리오 적합성** | ✅ 즉시 구현 가능 | △ GPU 서버 없으면 느림 |
| **파인튜닝** | 제한적 (OpenAI만) | ✅ 자유롭게 가능 |

**이 프로젝트 상황**

```
오픈소스 선택 시 현실적 문제:
  - GPU 서버 없으면 CPU 추론 → 응답 10~30초 (3초 목표 불가)
  - Ollama + llama3-8b 로컬 실행: 포트폴리오 데모에서 속도 문제
  - 한국어 품질: 7B~13B 모델은 Claude Haiku보다 낮음
  - Spring AI Ollama 연동은 가능하지만 불안정

독점 모델 선택 시:
  - 즉시 사용, 안정적인 한국어
  - 포트폴리오 데모에서 빠른 응답 보장
  - 비용: 포트폴리오 수준에서 무시 가능
```

**결정: 독점 모델 (Claude)**

```
오픈소스 선택하지 않는 이유:
  "빠른 검증 우선" 원칙 — 파이프라인 전체가 아직 미검증
  GPU 없이 오픈소스를 쓰면 속도 문제가 파이프라인 검증을 방해함
  파인튜닝도 파운데이션 모델 검증 후의 문제 (Step 3 임베딩과 동일 원칙)

오픈소스 전환 검토 시점:
  파이프라인 전체 검증 완료 후
  데이터 프라이버시 요구사항이 생길 때
  비용이 실제로 문제가 될 때
```

### Step 8. 컨텍스트 관련성 (Context Relevance)

> 검색된 컨텍스트가 사용자 질문에 실제로 필요한 정보를 담고 있는가?
> 측정 방법: **필요한 문장 수 / 전체 문장 수** = 관련성 점수

---

#### 관련성 수준 정의

| 수준 | 점수 범위 | 설명 | 조치 |
|------|----------|------|------|
| 높은 관련성 | 0.8 ~ 1.0 | 질문에 직접 답할 수 있는 정보 포함 | 정상 |
| 보통 관련성 | 0.4 ~ 0.7 | 관련 있지만 핵심 정보 누락 | 모니터링 |
| 낮은 관련성 | 0.0 ~ 0.3 | 질문과 관련 없는 내용 | 검색 전략 개선 필요 |

---

#### 이 프로젝트 적용 예시

**예시 1 — 높은 관련성 (정상 케이스)**

```
쿼리: "달달한 아이 간식 추천해줘"

검색된 컨텍스트:
  문장1: "꿀사과는 달콤하고 아삭한 사과로 아이 간식으로 추천합니다."  ← 필요 ✅
  문장2: "꿀사과는 1박스(5개) 단위, 12,000원입니다."              ← 필요 ✅
  문장3: "현재 재고 있습니다."                                   ← 필요 ✅

관련성 점수: 3/3 = 1.0  ← 높음, 정상
```

**예시 2 — 보통 관련성 (주의 필요)**

```
쿼리: "달달한 아이 간식 추천해줘"

검색된 컨텍스트:
  문장1: "꿀사과는 달콤하고 아삭한 사과입니다."                  ← 필요 ✅
  문장2: "꿀사과 산지는 경북 영주입니다."                       ← 불필요 ❌
  문장3: "꿀사과 포장은 친환경 재질을 사용합니다."               ← 불필요 ❌
  문장4: "12,000원, 재고 있음."                                ← 필요 ✅

관련성 점수: 2/4 = 0.5  ← 보통, 청킹 개선 검토
→ 원인: 상품 설명에 추천과 무관한 정보(산지, 포장)가 많이 포함됨
→ 개선: embeddingText 정제 시 산지/포장 정보 제거
```

**예시 3 — 낮은 관련성 (검색 전략 개선 필요)**

```
쿼리: "달달한 아이 간식 추천해줘"

검색된 컨텍스트:
  문장1: "제주 감귤은 제주도에서 재배합니다."                    ← 불필요 ❌
  문장2: "감귤 농가 직배송으로 신선도를 유지합니다."             ← 불필요 ❌
  문장3: "15,000원, 박스(10개) 단위."                         ← 불필요 ❌

관련성 점수: 0/3 = 0.0  ← 낮음, 검색 실패
→ 원인: "달달한", "아이 간식" 벡터가 감귤 설명과 유사하게 매핑됨
→ 개선 방향 (우선순위 순):
    1. embeddingText에 카테고리/태그 정보 강화
    2. 카테고리 메타데이터 사전 필터 적용
    3. 유사도 임계값 0.6 → 0.65로 상향
    4. HyDE 도입 검토 (Phase 2)
```

---

#### 관련성 점수 측정 방법

**방법 1 — LLM 기반 자동 평가 (권장)**

```
각 검색 결과 문장에 대해 LLM에게 판단 요청

프롬프트:
  "다음 질문에 답하기 위해 아래 문장이 필요합니까? yes/no로만 답하세요.
   질문: {query}
   문장: {sentence}"

→ yes 개수 / 전체 문장 수 = 관련성 점수
→ Spring AI로 배치 처리 가능
```

**방법 2 — 수동 샘플 테스트 (초기 검증용)**

```
구현 직후 수동으로 10~20개 쿼리 테스트:

테스트 쿼리 목록 (이 프로젝트 기준):
  1. "달달한 간식 추천해줘"
  2. "아이 선물로 뭐가 좋아요?"
  3. "만원 이하 과일 있어요?"
  4. "제주산 과일 추천해줘"
  5. "저번에 산 것과 비슷한 거 있어요?"

각 쿼리별 검색 결과를 직접 보고 관련성 점수 기록
→ 평균 0.7 미만이면 검색 전략 개선 후 재측정
```

---

#### 관련성 점수 기준 및 대응

```
목표: 평균 컨텍스트 관련성 점수 0.7 이상

점수별 대응:
  0.8 이상  → 정상, 다음 평가 단계(답변 충실성) 진행
  0.6~0.7   → embeddingText 정제 또는 청킹 개선
  0.4~0.6   → 메타데이터 필터 조건 강화, 유사도 임계값 상향
  0.4 미만  → 임베딩 텍스트 포맷 전면 재검토

낮은 점수의 주요 원인:
  1. embeddingText에 노이즈 정보 포함 (산지, 포장재 등)
     → Step 1 정제 규칙 강화
  2. 카테고리 필터 미적용
     → 쿼리에서 카테고리 추출 후 사전 필터 추가
  3. 유사도 임계값이 너무 낮음
     → 0.6 → 0.65~0.7로 상향
  4. 청크 크기 문제 (여러 정보가 섞인 경우)
     → 500토큰 제한 재검토
```

### Step 9. 답변 충실성 (Answer Faithfulness)

> 생성된 답변이 검색된 컨텍스트에만 근거하는가? 없는 내용을 만들어냈는가?
> 측정 방법: **뒷받침되는 주장 수 / 전체 주장 수** = 충실성 점수

---

#### 충실성 수준 정의

| 수준 | 점수 범위 | 설명 | 조치 |
|------|----------|------|------|
| 높은 충실성 | 0.8 ~ 1.0 | 컨텍스트 사실에만 근거, 환각 없음 | 정상 |
| 보통 충실성 | 0.5 ~ 0.7 | 일부 내용이 컨텍스트와 불일치 | 프롬프트 Grounding 강화 |
| 낮은 충실성 | 0.0 ~ 0.4 | 없는 상품, 가격, 재고를 만들어냄 | 즉시 개선 필요 |

---

#### 이 프로젝트 적용 예시

**예시 1 — 높은 충실성 (정상 케이스)**

```
컨텍스트:
  "꿀사과는 달콤하고 아삭한 사과로 아이 간식으로 추천합니다.
   1박스(5개) 단위, 12,000원, 재고 있음."

생성된 답변:
  "꿀사과(12,000원) 어떠세요? 달콤하고 아삭해서 아이 간식으로 딱이에요.
   1박스에 5개 들어있어요."

주장 분해:
  주장1: "꿀사과 12,000원"       → 컨텍스트 뒷받침 ✅
  주장2: "달콤하고 아삭함"        → 컨텍스트 뒷받침 ✅
  주장3: "아이 간식으로 추천"     → 컨텍스트 뒷받침 ✅
  주장4: "1박스 5개"             → 컨텍스트 뒷받침 ✅

충실성 점수: 4/4 = 1.0  ← 높음, 정상
```

**예시 2 — 낮은 충실성 (환각 발생)**

```
컨텍스트:
  "꿀사과는 달콤하고 아삭한 사과로 아이 간식으로 추천합니다.
   1박스(5개) 단위, 12,000원, 재고 있음."

생성된 답변:
  "꿀사과(12,000원) 추천드려요! 경북 영주 농가 직배송이라 신선해요.
   지금 구매하시면 사은품도 드려요. 오늘 마감 물량만 남았어요."

주장 분해:
  주장1: "꿀사과 12,000원"          → 컨텍스트 뒷받침 ✅
  주장2: "경북 영주 농가 직배송"     → 컨텍스트 없음 ❌ (환각)
  주장3: "신선함"                   → 컨텍스트 없음 ❌ (환각)
  주장4: "사은품 제공"              → 컨텍스트 없음 ❌ (환각)
  주장5: "오늘 마감 물량"           → 컨텍스트 없음 ❌ (환각, 위험)

충실성 점수: 1/5 = 0.2  ← 낮음, 즉시 개선 필요
→ 원인: Grounding 지시가 약하거나 없음
→ 이 케이스가 위험한 이유:
    "마감 물량" 같은 거짓 정보가 소비자 판단에 영향
    실제로 없는 사은품을 약속 → 신뢰 손상
```

**예시 3 — 재고 관련 환각 (이 프로젝트 특이 케이스)**

```
컨텍스트:
  "감귤, 15,000원, 박스(10개) 단위."
  (재고 정보 없음 → inStock 필드 누락된 경우)

생성된 답변:
  "감귤 있어요! 지금 바로 주문 가능합니다."

주장 분해:
  주장1: "감귤 있음"               → 컨텍스트 뒷받침 불가 ❌ (재고 정보 없음)
  주장2: "바로 주문 가능"          → 컨텍스트 뒷받침 불가 ❌

충실성 점수: 0/2 = 0.0
→ 원인: inStock 필드가 컨텍스트에 없으면 LLM이 임의로 "있다"고 답변
→ 대응: ES 저장 시 inStock 필드 필수, 재고 없으면 프롬프트에 아예 미포함
```

---

#### 충실성 점수 측정 방법

**LLM 기반 자동 평가**

```
Step 1 — 답변을 개별 주장으로 분해
  프롬프트:
    "아래 답변을 개별 사실 주장(claim)으로 분리하세요. JSON 배열로 반환하세요.
     답변: {answer}"
  → ["꿀사과 12,000원", "달콤하고 아삭함", "아이 간식 추천", "1박스 5개"]

Step 2 — 각 주장이 컨텍스트에서 뒷받침되는지 확인
  프롬프트:
    "아래 주장이 컨텍스트에 근거합니까? yes/no로만 답하세요.
     컨텍스트: {context}
     주장: {claim}"
  → yes/no 판정

Step 3 — 점수 계산
  충실성 점수 = yes 개수 / 전체 주장 수
```

---

#### 충실성 점수 기준 및 대응

```
목표: 평균 충실성 점수 0.8 이상

점수별 대응:
  0.8 이상  → 정상, 다음 평가 단계(답변 관련성) 진행
  0.6~0.8   → 시스템 프롬프트 Grounding 지시 강화
  0.4~0.6   → Few-Shot 예시에 "없는 내용 말하지 않기" 케이스 추가
  0.4 미만  → 환각이 심각, 컨텍스트 포맷 재검토

낮은 점수의 주요 원인과 대응:
  원인1: Grounding 지시가 약함
    → "반드시 아래 [상품 목록]에 있는 정보만 사용하세요" 강화
    → "목록에 없는 정보(산지, 배송 기간, 사은품 등)는 절대 언급 금지" 추가

  원인2: 재고/가격 정보 누락된 상품이 컨텍스트에 포함됨
    → ES 저장 시 inStock, price 필수 필드 검증 추가
    → null 값 상품은 색인 자체를 거부

  원인3: LLM이 "친절하게 보이려고" 정보를 추가함
    → Few-Shot 예시에 좋은 점원 = 모르면 솔직히 말하는 점원 명시
    → "확실하지 않은 정보는 말하지 마세요" 시스템 프롬프트에 추가
```

### Step 10. 답변 관련성 (Answer Relevance)

> 생성된 답변이 사용자의 질문에 직접적으로 답하는가? 불필요한 내용이 희석시키지 않는가?
> 측정 방법: **역방향 질문 생성** — 답변에서 역으로 질문을 만들어 원래 질문과 코사인 유사도 비교

---

#### 관련성 수준 정의

| 수준 | 점수 범위 | 설명 | 조치 |
|------|----------|------|------|
| 높은 관련성 | 0.8 ~ 1.0 | 질문에 직접 답변, 불필요한 내용 없음 | 정상 |
| 보통 관련성 | 0.5 ~ 0.7 | 답은 했지만 관련 없는 내용이 포함됨 | 응답 길이/범위 제한 |
| 낮은 관련성 | 0.0 ~ 0.4 | 질문을 벗어난 답변 | 페르소나/프롬프트 재검토 |

---

#### 측정 원리 (역방향 질문 생성)

```
원리:
  "좋은 답변은 역으로 유추한 질문이 원래 질문과 비슷해야 한다"

측정 흐름:
  원래 질문: "달달한 아이 간식 추천해줘"
       ↓
  생성된 답변 → LLM에게 전달
  "이 답변은 어떤 질문에 대한 답인가? 3개 생성하세요"
       ↓
  역방향 질문 3개 생성
       ↓
  각 역방향 질문과 원래 질문의 코사인 유사도 계산
       ↓
  평균 = 답변 관련성 점수
```

---

#### 이 프로젝트 적용 예시

**예시 1 — 높은 관련성 (정상 케이스)**

```
원래 질문: "달달한 아이 간식 추천해줘"

생성된 답변:
  "꿀사과(12,000원)랑 제주 감귤(15,000원) 있어요.
   아이 간식으로는 꿀사과가 달콤해서 인기 많아요. 어떤 걸로 할까요?"

역방향 질문 3개:
  Q1: "아이 간식으로 살 만한 달달한 과일 추천해줘"
  Q2: "달콤한 아이 간식 뭐 있어요?"
  Q3: "어린이 간식 과일 추천"

원래 질문과 코사인 유사도:
  Q1: 0.92  Q2: 0.88  Q3: 0.81
  평균: 0.87  ← 높음, 정상
```

**예시 2 — 낮은 관련성 (관련 없는 내용 포함)**

```
원래 질문: "달달한 아이 간식 추천해줘"

생성된 답변:
  "꿀사과(12,000원) 추천드려요. 참고로 저희 마트는 매일 오전 9시~오후 9시
   운영하고, 주문은 배송지 기준 다음날 도착합니다. 감사합니다!"

역방향 질문 3개:
  Q1: "아이 간식 추천해줘"
  Q2: "마트 운영 시간이 언제예요?"
  Q3: "배송은 언제 도착해요?"

원래 질문과 코사인 유사도:
  Q1: 0.85  Q2: 0.12  Q3: 0.09
  평균: 0.35  ← 낮음, 운영시간/배송 정보가 관련성 희석

→ 원인: LLM이 "친절하게" 관련 없는 정보를 덧붙임
→ 답변이 질문을 벗어나 마트 안내문이 됨
```

**예시 3 — 이 프로젝트 특이 케이스: 주문 흐름 이탈**

```
원래 질문: "감귤 주세요"

생성된 답변:
  "감귤은 현재 품절입니다. 이번 달 제철 과일로는 딸기, 한라봉,
   천혜향이 있으며, 산지별 가격 비교를 원하시면 알려드릴게요.
   또한 과일 꾸러미 정기배송 서비스도 있습니다."

역방향 질문 3개:
  Q1: "품절된 상품 대안 뭐 있어요?"

  Q3: "정기배송 서비스 있어요?"

  Q1: 0.41  Q2: 0.18  Q3: 0.11
  평균: 0.23  ← 매우 낮음

→ 이상적인 답변:
    "죄송해요, 감귤은 지금 품절이에요. 비슷한 달달한 과일로
     꿀사과(12,000원) 어떠세요?"  ← 1~2문장으로 끝
```

#### 관련성 점수 측정 구현

```java
// AnswerRelevanceEvaluator.java
public double evaluate(String originalQuery, String answer) {

    // Step 1: 답변으로부터 역방향 질문 3개 생성
    String reversePrompt = """
        아래 답변은 어떤 질문에 대한 답변인가요?
        가능한 질문 3개를 JSON 배열로 생성하세요.
        답변: %s
        """.formatted(answer);

    List<String> reverseQuestions = chatClient
        .prompt(reversePrompt)
        .call()
        .entity(new ParameterizedTypeReference<List<String>>() {});

    // Step 2: 원래 질문과 각 역방향 질문의 코사인 유사도 계산
    float[] originalVector = embeddingModel.embed(originalQuery);

    double avgSimilarity = reverseQuestions.stream()
        .mapToDouble(rq -> {
            float[] rqVector = embeddingModel.embed(rq);
            return cosineSimilarity(originalVector, rqVector);
        })
        .average()
        .orElse(0.0);

    return avgSimilarity;
}
```

---

#### 관련성 점수 기준 및 대응

```
목표: 평균 답변 관련성 점수 0.8 이상

점수별 대응:
  0.8 이상  → 정상, 세 지표(관련성/충실성/답변관련성) 모두 통과
  0.6~0.8   → 응답 길이 제한 강화 (최대 150토큰 → 100토큰)
             시스템 프롬프트: "질문에만 집중해서 1~2문장으로 답하세요" 추가
  0.4~0.6   → Few-Shot 예시에 "짧고 정확한 답변" 케이스 강화
  0.4 미만  → 페르소나 정의 재검토 ("마트 직원"이 아닌 다른 역할로 동작 중)

낮은 점수의 주요 원인:
  원인1: LLM이 관련 없는 정보를 자발적으로 추가
    → 시스템 프롬프트: "질문에 직접 관련된 내용만 답하세요"
    → "마트 운영 정책, 배송 안내는 질문하지 않으면 언급하지 마세요"

  원인2: 품절 시 대안을 지나치게 많이 나열
    → "대안은 1개만 추천하세요" 명시

  원인3: 답변이 너무 길어 질문의 핵심이 희석됨
    → max_tokens 제한 + 시스템 프롬프트 응답 길이 지시
```

#### 세 가지 평가 지표 관계

```
컨텍스트 관련성  → "검색이 잘 됐는가?"        색인/검색 파이프라인 품질
답변 충실성      → "없는 내용을 만들었는가?"   프롬프트 Grounding 품질
답변 관련성      → "질문에 집중해서 답했는가?" 페르소나/응답 제어 품질

세 지표가 모두 0.8 이상일 때:
  → RAG 파이프라인이 정상적으로 동작하고 있음
  → 구현 단계로 진행 가능

하나라도 0.6 미만이면:
  → 해당 단계(검색/프롬프트) 개선 후 재측정
```

### 평가 단계 유의사항

---

#### 1. 처음에는 인간 평가로 시작

```
Ground Truth 데이터셋 (정답 기준 데이터) 구축은 비용이 큼
→ 처음부터 자동화 평가 구축에 투자하면 배보다 배꼽이 커짐

현실적인 순서:
  1단계: 소수 쿼리(10~20개)로 직접 눈으로 품질 확인
          "달달한 간식 추천해줘" → 결과가 말이 되는가?
          검색된 상품이 실제로 관련 있는가?

  2단계: 품질이 어느 정도 나오면 자동화 평가 도입
          LLM 기반 측정 (컨텍스트 관련성, 충실성, 답변 관련성)

  3단계: 자동화 평가가 안정화되면 Ground Truth 데이터셋 구축 검토

이 프로젝트 적용:
  포트폴리오 수준 → 1단계(수동)로 충분
  서비스 운영 수준이 되면 2~3단계 도입
```

---

#### 2. 평가 지표 간 우선순위 설정

```
세 지표를 모두 완벽하게 달성하기는 어려움
→ 서비스 특성에 따라 우선순위 결정 필요

지표 충돌 예시:
  답변 충실성 ↑ (없는 말 하지 않음)
    vs
  답변 관련성 ↑ (질문에 집중, 짧게)

  → 품절 시 "없어요" 한마디가 충실성은 높지만
    사용자 입장에서는 대안 안내가 더 관련성 높게 느껴질 수 있음
```

**이 프로젝트 우선순위**

| 순위 | 지표 | 이유 |
|------|------|------|
| 1순위 | **답변 충실성** | 없는 상품/가격/재고를 말하면 주문 실패 → 신뢰 손상 |
| 2순위 | **답변 관련성** | 점원처럼 빠르고 핵심만 → UX 직결 |
| 3순위 | **컨텍스트 관련성** | 검색 품질은 백엔드 개선으로 점진적 향상 가능 |

```
목표 기준 (우선순위 반영):
  답변 충실성    0.85 이상  (엄격)
  답변 관련성    0.80 이상  (엄격)
  컨텍스트 관련성 0.70 이상  (보통)
```

#### 3. 파이프라인 변경 시 재평가 필수

```
아래 변경이 발생하면 세 지표 모두 재측정:

  변경 항목                 영향받는 지표
  ──────────────────────────────────────────
  임베딩 모델 교체           컨텍스트 관련성, 답변 충실성
  embeddingText 포맷 변경   컨텍스트 관련성
  청킹 단위 변경            컨텍스트 관련성
  시스템 프롬프트 수정       답변 충실성, 답변 관련성
  운영용 LLM 모델 업데이트  답변 충실성, 답변 관련성
  유사도 임계값 변경         컨텍스트 관련성

→ 작은 변경 하나가 전혀 예상치 못한 지표 하락을 만들 수 있음
→ 변경 전/후 수치를 반드시 기록해 두어야 "이 변경이 왜 나빠졌는지" 추적 가능
```

#### 4. 평가용 LLM과 운영용 LLM 분리

```
세 지표 모두 LLM이 판단에 관여
→ 운영용 LLM(Sonnet)으로 평가까지 하면 비용 2배

분리 전략:
  운영용: Claude Sonnet 4.6  → 실제 추천/주문 대화
  평가용: Claude Haiku 4.5   → 관련성/충실성 판단 (단순 yes/no, 질문 생성)
```

**비용 비교 (주문 1건 기준)**

```
평가 포함 총 LLM 호출 횟수 (대화 5턴 기준):
  운영 호출: 5회 (대화 턴당 1회)
  평가 호출: 15회 (턴당 컨텍스트 관련성 + 충실성 + 답변 관련성 각 1회)
  총 20회

운영/평가 모두 Sonnet:  20회 × Sonnet 비용 = 높음
운영 Sonnet + 평가 Haiku: 5회 × Sonnet + 15회 × Haiku = 약 60~70% 절감

→ 포트폴리오 수준에서도 평가용은 Haiku 사용 권장
```

**Spring AI 설정**

```java
// 운영용 — 추천/대화
@Bean("orderChatClient")
ChatClient orderChatClient(ChatModel sonnetModel) {
    return ChatClient.builder(sonnetModel).build();
}

// 평가용 — 관련성/충실성 판단
@Bean("evalChatClient")
ChatClient evalChatClient(ChatModel haikuModel) {
    return ChatClient.builder(haikuModel).build();
}
```

```yaml
# application.yml
spring:
  ai:
    anthropic:
      chat:
        options:
          model: claude-sonnet-4-6      # 운영용 기본값
      eval:
        model: claude-haiku-4-5-20251001 # 평가용
```

### 컨텍스트 관련성 요약

```
측정:   필요한 문장 수 / 전체 문장 수
목표:   평균 0.7 이상
방법:   초기 → 수동 샘플 20개 / 운영 → LLM 자동 평가
낮을 때: embeddingText 정제 → 메타데이터 필터 → 임계값 상향 순으로 개선
```
